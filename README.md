# 数模 AI 专用训练与比赛工作流

这个仓库用于训练一套面向数学建模竞赛的 AI 协作系统。它不是简单收集提示词，也不是让 AI 直接套模板写论文，而是通过优秀论文学习、提示词分层、旧题闭环测试和失败复盘，把 AI 逐步训练成一个更稳定的数模队友。

核心目标：

```text
优秀论文学习
-> 生成学习卡片和知识卡片
-> 沉淀 base / plugin / patch 三层提示词
-> 用旧题跑总控诊断
-> 按评分表自评和复盘
-> 局部修复提示词
-> 形成比赛可用的稳定工作流
```

比赛时的目标不是让 AI 一步到位给最终答案，而是让 AI 先做总控诊断、拆题、判断题型、列数据需求、比较模型、规划图表和人工确认项。路线确认后，再按模块让 AI 辅助建模、写代码、分析结果和写论文。

## 项目定位

本项目专门训练 AI 做三件事：

1. 读懂数学建模题，而不是直接套高级模型。
2. 把优秀论文经验转化成可复用的提示词规则。
3. 在比赛中按闸门推进，减少跑偏、乱用模型、缺数据、结果不可解释等问题。

当前重点题型是工程优化类，尤其是定日镜场、空间布局、资源配置、工程效率优化这类问题。后续可以继续扩展预测类、评价类、仿真类等插件。

## 目录结构

```text
prompt_base/
  基础总控提示词，适用于所有赛题。

prompt_plugins/
  题型插件，例如优化类、预测类、评价类等。

prompt_patches/
  从单篇优秀论文提炼出的经验补丁。

papers/
  优秀论文学习卡片和知识卡片 JSON。

tests/old_problems/
  旧题测试记录，用来验证提示词是否真的有效。

reviews/failure_cards/
  失败复盘卡，记录 AI 跑偏原因和修复方向。

prompt_versions/
  提示词版本日志，记录 draft、testing、stable、rollback 等状态。

output/closed_loop/
  自动化闭环测试产物，包括闭环摘要、重测任务、修复草案等。

training_log.md
  训练总日志，只记录每次训练或测试的关键事实。
```

说明：原始 PDF 论文和临时抽取文件不上传仓库，`.gitignore` 已忽略 `*.pdf` 和 `tmp/`。仓库只保存可复用的学习结果、提示词和测试记录。

## 三层提示词体系

本项目把提示词分成三层，避免把单篇论文经验污染成通用规则。

### 1. base：基础总控提示词

位置：`prompt_base/prompt_base_v1.0.md`

作用：

- 先读题、拆题、判断题型；
- 建立输入、处理、输出链路；
- 识别数据需求和人工确认项；
- 禁止一上来写论文、写代码或给最终答案。

所有赛题都先从 base 开始。

### 2. plugin：题型插件

位置示例：`prompt_plugins/plugin_optimization_v1.md`

作用：

- 针对某一类题补强硬规则；
- 优化题必须先写目标函数、决策变量、约束条件、评价指标；
- 使用遗传算法、粒子群、模拟退火前，必须先检查简单方法是否可行；
- 要求给出优化前后对比和敏感性分析。

当题目属于工程优化、布局优化、资源配置、路径规划、排班调度等场景时，叠加优化类 plugin。

### 3. patch：论文经验补丁

位置示例：

- `prompt_patches/patch_A092_engineering_optimization.md`
- `prompt_patches/patch_A127_engineering_layout_optimization.md`

作用：

- 保存某篇优秀论文中特别值得迁移的经验；
- 只迁移建模路线，不照搬公式、数值、布局和代码；
- 默认不进入 base，除非经过多题型反复验证。

例如 A092 的核心经验是：

```text
先建立基础评价模型
-> 再把后续问题转化为约束优化
```

这对工程优化题很有价值，但 A092 的定日镜公式、同心圆布局、具体数值不能直接照搬。

## 日常训练流程

日常训练用于赛前打磨 AI 工作流。推荐按下面顺序执行。

### S1. 学习一篇优秀论文

输入：

- 优秀论文 PDF；
- 当前训练工作流；
- 目标题型。

输出：

- `papers/年份_编号_学习卡片.md`
- `papers/年份_编号_知识卡片.json`
- 必要时生成一个 `prompt_patches/patch_xxx.md`

学习时重点提炼：

- 题目理解方式；
- 子问题拆解；
- 建模路线；
- 模型选择理由；
- 数据处理方式；
- 图表如何服务论证；
- 可迁移经验；
- 不可照搬内容。

### S2. 生成或更新题型插件

当多篇论文都指向同一种题型经验时，可以更新对应 plugin。

例如工程优化题必须检查：

- 是否先建立评价函数或目标函数；
- 是否明确优化目标；
- 是否明确决策变量；
- 是否明确约束条件；
- 是否比较简单方法和智能优化算法；
- 是否设计优化前后对比；
- 是否设计敏感性分析。

### S3. 用旧题做闭环测试

按照 `AI自动化闭环测试工作流.md` 执行：

```text
项目预检
-> 材料装载
-> 旧题任务建立
-> 总控诊断生成
-> 100 分评分
-> 失败标签检查
-> 闸门判定
-> 旧题测试记录
-> 必要时失败复盘
-> 必要时修复草案
-> 训练日志更新
-> 重测任务生成
```

每次旧题测试必须生成：

- 旧题测试记录；
- 闭环摘要；
- 训练日志记录；
- 如果失败，生成失败复盘；
- 如果需要改提示词，先生成修复草案，不直接覆盖正式提示词。

### S4. 按评分表判断是否通过

总控诊断满分 100：

| 项目 | 分值 |
|---|---:|
| 题目理解 | 15 |
| 子问题拆解 | 15 |
| 题型判断 | 10 |
| 模型候选与取舍 | 20 |
| 数据需求 | 10 |
| 建模路线 | 15 |
| 图表与结果计划 | 10 |
| 人工确认项 | 5 |

关键失败标签：

| 标签 | 含义 |
|---|---|
| P3 | 模型不匹配 |
| P4 | 高级模型滥用 |
| P5 | 数据需求不清 |

通过条件：

```text
总分 >= 80
且没有 P3 / P4 / P5
```

同一题型至少通过 2 道旧题，且输出结构稳定，才允许建议标记为 stable。

## 怎么让 AI 执行一次训练

可以把下面这段话交给 AI：

```text
你现在是数学建模提示词系统训练代理。

请先阅读：
1. 数模AI训练总控工作流v3.md
2. AI自动化闭环测试工作流.md
3. training_log.md
4. prompt_base/prompt_base_v1.0.md
5. 对应题型的 plugin
6. 对应论文经验 patch
7. papers/ 中的学习卡片和知识卡片
8. tests/old_problems/ 中的旧题材料或测试模板

你的任务不是写论文，也不是写代码，而是完成一轮旧题总控诊断闭环测试。

请输出：
- 项目预检结果；
- 本轮材料摘要；
- 闭环测试任务卡；
- 总控诊断；
- 100 分评分；
- P1-P10 失败标签检查；
- 闸门判定；
- 旧题测试记录；
- 必要时失败复盘；
- 必要时提示词修复草案；
- 必要时重测任务；
- training_log.md 更新建议。

硬规则：
- 不直接写完整论文；
- 不直接写求解代码；
- 不直接给最终答案；
- 不跳过总控诊断；
- 不把单篇论文经验直接写入 base；
- 不因一次通过就标记 stable。
```

## 比赛时怎么使用

比赛时不要直接把题目丢给 AI 让它“解出来”。推荐按闸门推进。

### 阶段 1：总控诊断

给 AI：

- 新赛题全文；
- `prompt_base/prompt_base_v1.0.md`；
- 对应题型 plugin；
- 可能相关的 patch；
- 当前可用数据或附件说明。

要求 AI 只做总控诊断：

```text
不要写论文。
不要写代码。
不要直接给最终答案。
先完成题目理解、子问题拆解、题型判断、模型候选、数据需求、图表计划和人工确认项。
```

人工检查：

- AI 是否读懂题；
- 子问题拆得是否对；
- 模型是否服务问题；
- 数据是否足够；
- 是否乱用高级模型；
- 是否需要简化路线。

### 阶段 2：确认建模路线

路线确认后，再让 AI 输出：

- 每一问的建模路线；
- 目标函数或评价函数；
- 决策变量；
- 约束条件；
- 输入数据表；
- 结果表结构；
- 图表清单；
- 代码模块拆分。

优化题尤其要先确认：

```text
评价函数是什么？
目标函数是什么？
决策变量是什么？
约束条件是什么？
简单方法能不能做？
为什么需要更复杂的算法？
```

### 阶段 3：分模块让 AI 辅助实现

不要一次让 AI 写完整工程。按模块推进：

1. 数据读取和清洗；
2. 基础评价函数；
3. 基线模型；
4. 优化模型；
5. 结果表；
6. 图表；
7. 敏感性分析；
8. 模型评价。

每一段代码都要让 AI 说明：

- 输入是什么；
- 输出是什么；
- 如何验证；
- 失败时怎么降级。

### 阶段 4：结果解释

结果出来后，先让 AI 分析结果，不急着写论文。

检查：

- 结果是否满足约束；
- 是否有优化前后对比；
- 是否有敏感性分析；
- 是否存在异常值；
- 模型结论能不能用现实语言解释；
- 图表是否服务结论。

### 阶段 5：论文写作

只有当前面路线、代码、结果和图表都通过后，才让 AI 写论文。

推荐顺序：

1. 问题重述；
2. 问题分析；
3. 模型假设；
4. 符号说明；
5. 模型建立；
6. 模型求解；
7. 结果分析；
8. 模型检验；
9. 模型评价；
10. 摘要。

摘要建议最后写，因为它必须基于最终模型和结果。

### 阶段 6：审稿和反查

让 AI 站在评委角度审稿：

```text
请按数学建模竞赛评审视角检查本文。
重点找：
1. 题目理解是否偏；
2. 模型是否不匹配；
3. 是否滥用高级算法；
4. 数据是否支撑模型；
5. 约束是否漏掉；
6. 结果是否只列数字没有解释；
7. 图表是否服务论证；
8. 摘要是否空泛；
9. 是否有明显扣分点。
```

审稿未通过，不交最终稿。

## 工程优化题比赛提示词组合

当比赛题明显属于工程优化、布局优化、资源配置或参数设计时，推荐组合：

```text
prompt_base/prompt_base_v1.0.md
+ prompt_plugins/plugin_optimization_v1.md
+ prompt_patches/patch_A092_engineering_optimization.md
+ 视情况加入 prompt_patches/patch_A127_engineering_layout_optimization.md
```

其中：

- A092 patch 负责“评价函数先行”；
- A127 patch 负责“空间布局、分区、规则化、变步长搜索”；
- plugin 负责“目标、变量、约束、算法质疑、敏感性分析”；
- base 负责“总控诊断、输入输出链、人工确认项”。

## 使用原则

1. 先诊断，后建模。
2. 先评价函数，后优化算法。
3. 先简单方法，后高级模型。
4. 先结果解释，后论文写作。
5. 先失败复盘，后提示词修复。
6. 没有人工确认，不进入下一阶段。
7. 没有两道旧题通过，不标记 stable。
