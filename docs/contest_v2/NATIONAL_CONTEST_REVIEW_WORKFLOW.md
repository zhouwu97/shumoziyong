# 全国大学生数学建模竞赛标准独立评审工作流

## 1. 定位

本工作流用于 Contest Production v2 的论文终审。评审目标不是只确认文件、数字和程序一致，而是判断提交物是否达到全国大学生数学建模竞赛参赛论文应有的建模与论证质量。

本标准由仓库内优秀论文学习材料抽象得到，主要来源为：

- `papers/2023_A092_学习卡片.md` 与 `papers/2023_A092_知识卡片.json`；
- `papers/2023_A127_学习卡片.md` 与 `papers/2023_A127_知识卡片.json`；
- `papers/2023_B311_学习卡片.md` 与 `papers/2023_B311_知识卡片.json`；
- `papers/2023_B477_学习卡片.md` 与 `papers/2023_B477_知识卡片.json`。

来源、核验状态、允许用途和规则晋级状态统一记录在 `papers/EXCELLENT_PAPER_REVIEW_STANDARD_REGISTRY.json`。其中 A092 的逐条 claims 已标记为 verified；其他卡片中的经验作为审稿启发使用，不作为事实真值。本工作流是内部操作性标尺，不声称是组委会公布的官方评分细则，也不根据同题优秀论文的最终答案反向对答案。

## 2. 新优秀论文的增量学习

每学习一篇新优秀论文，必须实际写入学习卡片、知识卡片、patch 索引、训练日志和优秀论文评审标准注册表。学习资产允许自动落盘，但评审规则按以下状态晋级：

```text
single_paper_candidate
→ cross_paper_candidate
→ validated_candidate
→ global_active
```

单篇、未核验或与具体题目强绑定的经验不得自动进入全局 Reviewer 硬规则。只有已核验且跨论文重复出现，或经过独立旧题验证的机制，才可升级。升级时必须记录来源、证据、适用范围、误用风险、版本和变更理由；冲突规则不得静默覆盖。

写入前必须做语义去重；已有同义规则时只补证据、适用范围、反例或冲突，不新建重复 patch。`validated_candidate` 只能由无同题泄漏的 `production_replay` 或陌生题完整生产支持，diagnostic、Prompt 回归、文件生成或旧 Gate 通过不构成生产证据。学习库同时登记“复杂模型不如基线”“图表多但无论证”等反例，避免只积累成功外观。

评审新题时只使用注册表中允许用于通用评审的规则，不读取同题优秀论文答案。具体执行见 `docs/workflows/01_论文学习流.md`。

## 3. 强制独立性与自动触发

每轮 Final Review 必须创建一个新的 Codex 桌面对话，不得由论文作者任务自评，也不得在原 Reviewer 对话中直接复审。

新题完成工程闭环后，必须先通过 `docs/workflows/03_新题执行流.md` 定义的 Paper Admission。只有 `ENGINEERING_VERIFICATION=PASS`、`PAPER_ADMISSION=PASS`、`paper_type=submission_candidate` 且准入记录绑定当前 PDF 摘要时，作者任务才能自动构建隔离交接包并调用 Codex 桌面的新任务接口启动 Reviewer。工程 PASS 或 `0 ERROR / 0 WARNING` 不能跳过论文准入。

Reviewer 完成后，作者任务只集中修补 MUST；重建受影响的计算、Verification、Ledger、论文、Paper Admission 与 package 后，必须再自动创建另一个全新的 Reviewer 任务复审，禁止复用上一轮对话。

这是 Agent/桌面运行时检查点，不属于 `contest` 薄 CLI 的职责。若当前环境没有创建新任务的能力，作者任务只能停在“交接包已完成”的外部检查点并明确报告，不能在原任务内冒充独立评审。

Reviewer 只读取隔离交接目录中的最终产物和本工作流快照，不读取：

- 作者对话、中间推理和修补清单；
- 旧论文、旧结果、旧 Gate；
- 同题优秀论文、同题题解、参考答案或同题专用代码；
- 上一轮评阅文件。

修补后的复审必须再次新建对话，防止 Reviewer 被自己的旧结论锚定。

## 4. 优秀论文画像

评审必须从以下共同特征判断论文，而不是只看 Verification PASS：

1. 题目被转换成清楚的“现实对象—数学变量—评价指标—约束—输出”链路。
2. 子问题之间有继承关系：前问建立评价模型或基础机制，后问复用并扩展，而非三段互不相干的计算。
3. 模型选择有题目机制依据；复杂算法不是装饰，算法输入、输出和停止条件可解释。
4. 优化前先固定统一评价口径，并给出简单、可复算的基线；优化结果必须在同一口径下比较。
5. 可信度来自公式推导、特殊情形、边界检查、独立复算、基线对照、稳定性或敏感性，而不是一次运行或自称最优。
6. 图表按推理顺序服务论证：机制图解释模型，结果表给数值，对比图说明增益，空间或路径图展示方案，敏感性图说明边界。
7. 摘要按问题给出方法、核心结果和结论；正文公式、变量、单位、参数和结果能够独立阅读。
8. 降维、分区、规则化或代理模型必须说明适用依据和排除的搜索空间；局部或受限最优不得写成全局最优。
9. 若新方案牺牲某个指标，必须解释换来了什么目标收益、约束满足或工程价值。
10. 局限性针对本模型的误差来源、样本、参数、求解界和推广范围，不能只写空泛套话。

## 5. 100 分操作性评分与判定门槛

该分值只用于内部质量比较，不对应官方奖项或资格。

| 维度 | 分值 | 核心问题 |
|---|---:|---|
| 题目覆盖与最终回答 | 20 | 是否逐问作答；每问是否给出题目要求的明确决策、数值、方案或结论 |
| 数学模型正确性与针对性 | 20 | 定义、假设、变量、目标、约束、随机/统计/机理关系是否正确且针对题意 |
| 求解、验证与证据 | 20 | 算法是否适配；基线、边界、特殊情形、收敛界、多次运行或独立验证是否充分 |
| 结果解释与决策价值 | 15 | 是否解释结果由何而来、改变了什么、取舍是什么以及是否具有实际意义 |
| 论文结构、图表与可读性 | 15 | 摘要、正文、公式、引用、图表和结论是否形成自洽、可独立阅读的论证链 |
| 创新性与非通用贡献 | 10 | 是否有题目机制驱动、相对基线且被证据支持的有效贡献；是否避免算法堆砌 |

Reviewer 必须逐项给分、给证据页码或 Ledger/表格键，并输出总分。仅有总分而无证据视为评审不完整。分数不能抵消 MUST 或 Paper Admission 缺口。

### 5.1 建议提交的必要条件

```text
总分 >= 80
且 MUST 为空
且题目覆盖与最终回答 >= 14/20
且数学模型正确性与针对性 >= 14/20
且求解、验证与证据 >= 14/20
且每个 15 分维度 >= 9/15
且创新性与非通用贡献 >= 6/10
且 Paper Admission 对当前 PDF 有效
```

创新性较弱会限制竞争力，但创新维度本身不是一票否决项；正确、完整、证据充分但创新有限的论文仍可被建议提交。

### 5.2 Reviewer 结论

| 结论 | 条件 |
|---|---|
| `SUBMISSION_RECOMMENDED` | 总分和全部单项门槛满足、MUST 为空、无根本性缺陷 |
| `MAJOR_REVISION` | 70–79 分，或总分虽高但存在可修复 MUST/单项未达线 |
| `NOT_RECOMMENDED` | 总分低于 70，或存在需要重建核心模型/验证的根本性缺陷、无效数据链或严重漏答 |

若总分区间与硬缺陷冲突，以更严格结论为准。Reviewer 不得为了让平均分“好看”而提高存在硬伤的其他维度。

### 三项独立质量审计

六维总分之外还必须完成三项审计，不得用一个“总体印象分”替代：

1. **创新性审计**：列出基线、实际新增机制、为何适配题意、量化增益或验证证据、适用边界。只换算法名、堆模型、增加流程层级或无对照的复杂化不算创新。
2. **可读性审计**：检查摘要是否逐问交代方法和结果，变量是否首次出现即定义，段落是否各有论证目的，图题/表题能否自解释，长句密度、术语跳跃、内部工程语言和结论重复是否影响阅读。
3. **AI 痕迹风险审计**：只输出 `低/中/高`、命中证据数、具体页码/句子类型和修改建议。禁止输出“AI 生成概率”“AI率百分比”或把检测器分数当事实，因为这类数字不能可靠证明作者身份。

AI 痕迹风险重点检查成簇出现的证据：空泛拔高、宣传式措辞、无数字支撑的“实验表明显著”、算法名堆砌、反复的“首先—其次—最后”、机械三段式、同义改写循环、句式节奏过度一致、泛化评价与局限、过密小标题或粗体、作者任务/提示词/Gate 等内部生产语言、虚构引用以及无来源数字。正式语法、数学结构、必要术语或单个常见过渡词本身不能作为 AI 证据。

作者在构建交接包前必须执行一次学术文本去模板化检查：保留公式、数据、引用、专名和真实技术语气，只改写上述模板化表达。任何改写后都要重新编译 PDF、重建 package，并确认正文数字与 Ledger 仍一致。

## 6. 不受总分抵消的阻断项

以下任一项成立，必须修复列表不得为空：

- 漏答任一子问题，或缺少题目强制附件/结果；
- 论文不能独立说明核心模型，主要模型只存在于代码或检查器中；
- 关键定义、公式、单位、目标或约束与题意矛盾；
- 关键数字无法复算，或正文、结果、图表、附件相互矛盾；
- 优化题只有旧方案评价，没有形成可信的重新规划或足够的最优性/搜索证据；
- 求解 gap、样本量、拟合误差或启发式不稳定性足以推翻核心结论，却仅用“预算有限”解释；
- 图表误导、退化或与正文结论冲突，导致方案效果无法判断；
- 用通用流程、内部系统语言或算法名堆砌代替实际建模；
- 将局部、受限或随机对照结果夸大为全局最优、普遍规律或奖项水平。
- 正文残留作者任务、提示词、Reviewer、Gate 或其他内部生产文本，或存在虚构引用、无来源关键数字。
- 论文实质是扩展摘要、结果图集或技术报告，任一核心模型无法脱离代码独立审阅；
- 训练/验证、候选/最终评价或特征/标签处理存在足以使结果失真的数据泄漏或无效验证；
- 求解质量不足以支持“无更优方案”“旧方案稳健”或类似强结论；
- Paper Admission 记录缺失、为 FAIL/PENDING，或其 PDF 摘要与被评 PDF 不一致。

一般性的语言模板化只进入建议修复；只有它损害可读性、掩盖证据、造成虚假陈述或暴露内部生产过程时才成为 MUST。

## 7. Reviewer 输出合同

评阅报告必须包含：

1. 竞赛标准总评；
2. 六维评分表与逐项证据；
3. 与优秀论文画像的差距；
4. 必须修复；
5. 建议修复；
6. 可接受风险；
7. 模型正确性；
8. 求解与结果可信性；
9. 图表与论文表达；
10. 完整性和提交风险；
11. 是否建议提交；
12. 仅作内部判断的“优秀论文竞争力”评价及不确定性说明。
13. 创新证据表：基线、贡献、题意依据、量化证据和适用边界；
14. 可读性缺陷清单：页码、问题、对理解的影响和修改建议；
15. AI 痕迹风险：低/中/高、命中证据数、证据位置和修改建议，并声明它不是作者身份或 AI 生成概率判断。
16. 最终结构化结论：`SUBMISSION_RECOMMENDED / MAJOR_REVISION / NOT_RECOMMENDED`，并逐条核对总分、单项下限和 MUST 条件。

“必须修复为空”必须同时满足：阻断项全部排除、每问完整、模型可由论文独立审阅、核心结果有足够证据。Verification 0 error/0 warning 不能单独推出该结论。页数不设硬门槛；内容覆盖和论证闭合才是判据。

## 8. 工作流顺序

```text
完整计算与论文
→ Verification / Ledger / PDF / package
→ ENGINEERING_VERIFICATION，仅证明工程闭环
→ 作者侧创新证据、可读性与 AI 痕迹风险自检
→ 改写后重新编译、校验数字并重建 package
→ 逐问 Paper Admission
→ FAIL 时标为 technical_report 并返回大修
→ 构建隔离交接包并写入本标准快照
→ 自动新建独立 Reviewer 对话
→ 按竞赛标准评审
→ 作者任务集中修补 MUST
→ 重建所有受影响派生产物并重新 Paper Admission
→ 自动再新建一个独立 Reviewer 对话复审
→ 复审推荐、MUST 为空且文件摘要一致后派生 SUBMISSION_STATUS=READY
```
