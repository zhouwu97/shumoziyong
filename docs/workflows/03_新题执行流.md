# 新题执行流

## 1. 定位

本流程用于比赛当天和模拟赛。目标是连续产出数学上可辩护、代码可复现、论文可审阅、文件可提交的完整解答。

必须把四种结论彻底分开：

```text
ENGINEERING_VERIFICATION
→ PAPER_ADMISSION
→ EXTERNAL_REVIEW
→ SUBMISSION_STATUS
```

`verify`、PDF 编译和 package 完成只证明工程闭环，不证明论文质量。普通新题默认采用 `standard`；只有能力评测、历史争议复现或用户明确要求封存时才采用 `formal_audit`。

## 2. 最小目录与数值真源

```text
题目目录/
├─ problem/                              # 官方题面、附件、输出模板
├─ plan.md
├─ contest.json
├─ questions/<qid>/
│  ├─ question.json                     # 本问完整配置真源
│  ├─ model.md
│  ├─ run.py
│  ├─ results/
│  │  ├─ result.json                    # 唯一计算真源
│  │  └─ verification.json              # 独立检查器派生
│  ├─ figures/
│  ├─ paper.typ
│  └─ check.md
├─ result_ledger.json                   # 当前结果派生快照
├─ paper/
│  ├─ generated/results.typ             # Ledger 自动生成
│  ├─ main.typ 或 main.tex              # 唯一论文源稿入口
│  └─ submission.pdf
├─ reports/
│  ├─ ANALYSIS_MODELING_REPORT.md
│  ├─ RESULTS_REPORT.md                 # 人类解释，不是数值真源
│  ├─ LEARNING_CONTEXT.md               # 学习资产选择与实际落点
│  ├─ learning_context.json             # 机器可读学习上下文
│  ├─ ENGINEERING_VERIFICATION_REPORT.md
│  └─ PAPER_ADMISSION_REPORT.md
├─ review/paper_admission.json          # PDF 摘要绑定的准入结论
└─ package/
```

数值真源固定为：

```text
questions/<qid>/results/result.json
→ verification.json
→ result_ledger.json
→ paper/generated/results.typ
→ paper/main.typ 与各问 paper.typ
```

`RESULTS_REPORT.md` 只解释结果、误差、约束和可写结论。论文关键数字不得以它为真源，也不得手抄绕过 `results.typ`。

## 3. 生产主线

### 3.0 题型识别与学习资产检索

在建立模型前，作者任务必须读取 `papers/EXCELLENT_PAPER_REVIEW_STANDARD_REGISTRY.json`，生成 `reports/LEARNING_CONTEXT.md` 与 `reports/learning_context.json`。使用 `docs/contest_v2/LEARNING_CONTEXT_TEMPLATE.json`，只加载：

- 与当前题型和风险匹配的 `global_active` 规则；
- 至多 3 个注册表中已核验且允许跨题使用的模式。

不得一次性加载全部学习卡，不得读取同题优秀论文、同题题解或答案，不得复制历史论文的具体数值、公式、代码和结论。每个被选资产必须说明选择理由和计划用于哪一问、哪一类模型或哪一段论证；没有生产验证证据的候选规则不得进入作者上下文。

在首次编译完整稿前，`section_coverage_plan` 必须为每问准备以下六类材料，并全部标为 `READY`：

```text
题目回应
模型公式组
求解算法
结果表或图
验证
解释与边界
```

任何一项未准备，不得把短稿当成完整稿编译。成稿后必须填写 `application_record`：逐项记录学习资产是否采用、实际论文落点；未采用时写明证据或题型原因。学习上下文不是装饰性清单，其当前摘要必须绑定到 Paper Admission。

### 3.1 分析与建模

读取全部官方材料，完成：

- 子问题、输入输出、单位、评分目标和最终交付；
- 原始数据检查和题目专属矛盾；
- 默认建立 `1 个主模型 + 1 个廉价基线`，说明选择理由；
- 只有路线争议、主模型失败、验证暴露结构缺陷或贡献不足时，才增加结构备选；
- 变量、假设、公式、约束、算法、停止条件、基线和验证方法；
- 论文所需图表及其要支撑的具体结论。

将结论写入 `reports/ANALYSIS_MODELING_REPORT.md`。只有题面缺失、关键歧义会改变数学问题，或路线取舍涉及必须由用户决定的偏好时才暂停。

### 3.2 编码、实验与图表

根据建模报告实现并运行代码。每问写入自己的 Result、表格和图表，再由 Verifier 生成 Verification。最低证据按题型选择：

- 预测类：无数据泄漏的训练/验证或留出实验、廉价基线、误差分布和外推边界；
- 优化类：可行性、硬约束、基线、搜索覆盖、停止原因、incumbent/bound/gap 或等价质量证据；
- 仿真/机理类：量纲、初边值、数值稳定性、守恒/边界和参数敏感性；
- 评价类：指标定义、方向、归一化、权重依据、基线和排序稳定性。

若最终复用旧方案或声称“未找到更优方案”，必须证明候选生成与搜索质量足以支撑该结论。巨大 gap、过窄支持集或单次短时运行时，只能写“本次搜索未发现”，不能把求解不足包装成稳健决策。

不要把候选程序复制一遍称为“独立验证”；相同数据和假设下只能称复算一致性。

### 3.3 完整论文写作

论文使用项目现有 Typst 模板或比赛指定模板。`paper/main.typ` 或 `paper/main.tex` 是唯一内容源，必须包含全部问级正文；禁止用短 PDF、结果图集或 Markdown 摘要冒充论文。

正文至少覆盖摘要、问题重述与分析、假设、符号、数据处理、每问模型建立与求解、结果解释与检验、敏感性/不确定性、模型评价、结论和参考文献。每问按以下论证链成文：

```text
题目要求 → 明确回答 → 变量参数 → 问题专属数学表达
→ 求解/推导 → 核心结果 → 验证/对照 → 解释 → 适用边界
```

图表必须支撑比较、诊断、机制、权衡或决策。相同方案相减为零、完全重合曲线、内部候选名、流水线状态和纯验证痕迹不应占用正文版面。

成稿后做创新性、可读性与 AI 痕迹风险自检，但这三项不能替代下面的 Paper Admission。AI 痕迹只报告低/中/高与可定位证据，不生成“AI率”。任何文字改写后重新编译并复查数字、引用和附件。

## 4. 阶段一：工程验收

作者任务运行 Verification、Ledger、Typst、PDF 和 package，并写 `reports/ENGINEERING_VERIFICATION_REPORT.md`。它只回答：

- 代码能否运行；
- Result 与附件能否解析和复算；
- 所需检查是否全部执行且通过；
- Ledger 与 `results.typ` 是否为当前版本；
- PDF 能否编译并通过基本视觉检查；
- package 是否含核心文件且排除禁入文件。

输出只能是：

```text
ENGINEERING_VERIFICATION: PASS | FAIL
```

即使是 `0 ERROR / 0 WARNING`，也不得写成“论文通过”“全链路通过”或“可以提交”。

## 5. 阶段二：Paper Admission

工程验收 PASS 后，作者任务必须逐问填写 `reports/PAPER_ADMISSION_REPORT.md`，并将结果连同当前 PDF SHA-256、`learning_context.json` 路径与摘要写入 `review/paper_admission.json`。Paper Admission 是交给独立 Reviewer 前的内容准入，不是竞赛评分。

报告格式使用 `docs/contest_v2/PAPER_ADMISSION_TEMPLATE.md`，不得自行删减矩阵项目。

### 5.1 每问准入矩阵

| 检查项 | 要求 | 证据定位 |
|---|---|---|
| 题目具体要求 | 必需 | 题面要求与正文回应位置一一对应 |
| 明确的最终回答或决策 | 必需 | 摘要、结果节或正式附件 |
| 变量和参数定义 | 必需 | 符号表与首次定义位置 |
| 问题专属数学表达 | 必需 | 目标、约束、统计/机理关系及变量域 |
| 求解算法或推导过程 | 必需 | 算法步骤、停止条件和关键参数 |
| 核心数值结果 | 必需 | `results.typ` 变量与正文位置 |
| 基线或对照 | 依题型 | 主张增益、稳健性或优越性时必需 |
| 有效性/误差/约束检查 | 必需 | Verification 之外还要解释检查意义 |
| 图表或表格证据 | 必需 | 必须有决策或论证价值 |
| 结果解释 | 必需 | 回答“为什么、改变了什么、是否有实际意义” |
| 适用边界和局限 | 必需 | 与本模型和当前证据对应 |

每个格子必须标为 `PASS / PARTIAL / MISSING / NOT_APPLICABLE` 并给页码、公式号、图表号或 Ledger 键。`NOT_APPLICABLE` 必须说明题型理由；空泛地写“正文已有”不算证据。

交接构建器必须机器验证完整 JSON，而不是只相信顶层状态：

- 全部 11 个固定检查项存在；
- 除条件基线项外，全部必需项为 `PASS`；
- `PARTIAL/MISSING` 一律拒绝；
- 条件项 `NOT_APPLICABLE` 必须有非空理由；
- 每项 `evidence` 至少含一个可定位字符串；
- `direct_blockers` 为空；
- 所有必答问题都有矩阵；
- 学习上下文版本、选择权限、章节覆盖、应用回填和摘要绑定有效；
- PDF 摘要有效。

### 5.2 直接不准入项

以下任一项成立：

```text
paper_type = technical_report
PAPER_ADMISSION = FAIL
```

- 任一必答问题缺少明确最终回答；
- 任一核心模型没有足以独立审阅的数学表达；
- 只有结果图或总指标，没有模型建立、求解或结果解释；
- 只有可行解，却用搜索不足支撑“没有更优方案”或强决策结论；
- 同一方案被反复评价，却没有可信候选搜索或新的决策分析价值；
- 关键论证仍使用 Result、Verification、Ledger、Typst、Gate、Reviewer 等内部流程语言；
- 大量图表只是验证痕迹、退化对照或同一方案自比较；
- 论文主要模型不能脱离代码、检查器或交接说明理解；
- 数据泄漏、无效验证、虚构引用或无来源关键数字；
- 求解质量不足、误差/样本/边界足以推翻核心结论，却仍作强结论。

页数不是硬门槛。短论文只要内容闭合可以准入；加空话、放大图片或堆公式不能修复准入失败。

### 5.3 准入结论

全部必需项 PASS、条件项处理合理且无直接不准入项时：

```text
paper_type = submission_candidate
PAPER_ADMISSION = PASS
```

否则为 `technical_report / FAIL`，必须先大修，不能构建 Reviewer 交接包。准入文件中的 PDF 摘要与当前 PDF 不一致时，状态自动视为 `PENDING`。

## 6. 阶段三：独立 Reviewer

只有同时满足：

```text
ENGINEERING_VERIFICATION = PASS
PAPER_ADMISSION = PASS
paper_type = submission_candidate
paper_admission.pdf_sha256 = 当前 PDF 摘要
paper_admission.learning_context_sha256 = 当前学习上下文摘要
全部逐问矩阵和学习上下文机器校验通过
```

作者任务才自动构建隔离交接包并新建 Codex Reviewer 任务。Reviewer 使用 `docs/contest_v2/NATIONAL_CONTEST_REVIEW_WORKFLOW.md`，不得用工程 PASS 推出论文质量结论。

Reviewer 输出：

```text
EXTERNAL_REVIEW: SUBMISSION_RECOMMENDED | MAJOR_REVISION | NOT_RECOMMENDED
```

出现 MUST 时作者集中修补，随后重新运行受影响计算、Verification、Ledger、论文编译、Paper Admission 和 package，再创建一个全新的 Reviewer 任务复审。

## 7. 阶段四：提交就绪

作者任务无权自行声明 `SUBMISSION_STATUS: READY`。只有编排层确认以下条件同时成立时才能派生 READY：

- 工程验收 PASS；
- Paper Admission PASS 且绑定当前 PDF；
- 全新 Final Reviewer 对当前 PDF 给出 `SUBMISSION_RECOMMENDED`；
- MUST 为空；
- 被评 PDF、最终 package PDF 与复审记录中的 SHA-256 完全一致；
- 最终附件完成独立核包。

任何计算、正文、图表或附件在复审后改变，Reviewer 结论立即 stale，状态回到 NOT_READY。

统一状态输出为：

```text
ENGINEERING_VERIFICATION: PASS | FAIL
PAPER_TYPE: technical_report | submission_candidate
PAPER_ADMISSION: PASS | FAIL | PENDING
EXTERNAL_REVIEW: PENDING | MAJOR_REVISION | NOT_RECOMMENDED | SUBMISSION_RECOMMENDED
SUBMISSION_STATUS: NOT_READY | READY
```

## 8. 运行模式

| 模式 | 用途 | 额外要求 |
|---|---|---|
| `standard` | 普通比赛与日常模拟 | 完整生产、工程验收、Paper Admission、独立终审与全新复审 |
| `emergency` | 临近截止时间 | 减少结构备选和修订轮次；不得取消数学硬检查、Paper Admission 或最终人工核包 |
| `formal_audit` | 能力评测、争议复现、长期封存 | 在 standard 上增加输入冻结、独立 Validator、Formal Result 和必要的 Manifest/Seal |

`formal_audit` 不能自动提高模型或论文质量。普通比赛不得因仓库存在旧 Formal 工具而默认启用。

## 9. Codex 桌面版入口

```text
请执行 docs/workflows/03_新题执行流.md，模式 standard。

读取 problem/ 中全部官方材料，连续完成分析建模、代码实验、问级 Result/Verification、Ledger、
完整论文、工程验收和 Paper Admission。默认采用一个主模型加一个廉价基线；仅在证据需要时增加
结构备选。

建模前先从优秀论文注册表生成 LEARNING_CONTEXT，只选择适用的 global_active 规则和少量已核验
跨题模式；禁止读取同题答案。完整稿编译前完成每问章节覆盖计划，成稿后回填学习资产实际落点。

ENGINEERING_VERIFICATION 通过不等于论文通过。只有 PAPER_ADMISSION=PASS 且绑定当前 PDF 时，
才能自动新建独立 Reviewer 任务。修补 MUST 后必须重建所有受影响产物、重新准入，并创建另一
全新 Reviewer 任务复审。作者任务不得自行声明 SUBMISSION_STATUS=READY。
```

## 10. 可信边界

- 拟合误差不等于外部预测能力；
- 复算一致不等于模型有效；
- 文件存在不等于论文准入；
- Paper Admission 不等于 Reviewer 推荐；
- Reviewer 推荐若未绑定最终文件摘要，不等于提交就绪；
- 独立 AI 评阅不等于真人评阅；
- 单题成功不等于稳定的陌生题能力。
