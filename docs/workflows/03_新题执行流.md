# 新题执行流

## 使用场景

比赛当天或模拟赛当天使用。新题执行共享 Gate 0-5 的身份绑定、业务产物和机器校验，但不生成旧题的评分、失败标签、Patch 建议或首级晋级证据。

普通用户只需按[AI 工作目录固定入口](../entrypoints/AI_WORKSPACE_BOOTSTRAP.md)把官方材料放入
`problem/` 并启动 AI。比赛引擎必须锁定 approved tag 或 allowlisted commit，初始化后不追随
最新 main，也不依赖实时网络。

## Gate 0-5 定义

| Gate | 名称 | 核心产物 |
|---|---|---|
| Gate 0 | 题目与材料诊断 | `diagnosis.md`、`diagnosis.json`、材料缺失清单 |
| Gate 1 | 模型路线 | 已确认的主路线、备选路线与约束来源 |
| Gate 2 | 代码计划 | 模块、输入输出、验证与降级计划 |
| Gate 3 | 结果确认 | 可追溯结果、约束检查和稳健性结论 |
| Gate 4 | 论文确认 | 可写入论文的材料与叙述边界 |
| Gate 5 | 最终验收 | 评委视角审查、可复现性和扣分风险 |

## 运行模式

启动时必须显式选择模式；未指定时使用 `standard`。AI/Agent 负责 Gate 0-4 的建模、验证、
论文修订与可审计记录；Gate 5 的最终通过只能由人工决定。模式只决定 AI 执行的核验强度，
不取消路线确认和结果合理性确认。

| 模式 | 适用场景 | 强制确认点 |
|---|---|---|
| `strict` | 训练、赛前模拟、时间充足 | AI 完整执行 Gate 0-4；人工只在 Gate 5 最终决策 |
| `standard` | 普通比赛 | AI 执行 Gate 0-4；人工只在 Gate 5 最终决策 |
| `emergency` | 比赛后期时间不足 | AI 执行 Gate 0-4；人工只在 Gate 5 最终决策 |

`emergency` 只减少人工暂停点：不得跳过任何业务产物、Schema、哈希、数学或复现检查。

## Gate 0 入口提示词

```text
你现在执行“新题执行流”，运行模式为 standard。

请读取：
1. docs/workflows/03_新题执行流.md
2. prompt_base/prompt_base_v1.0.md
3. 当前题目最相关的 plugin
4. 当前题目可能相关的 patch
5. checklists/gate_0_material_diagnosis.md

本轮只做 Gate 0：题目与材料诊断。

禁止：
1. 不写完整论文；
2. 不写代码；
3. 不给最终答案；
4. 不直接套高级模型。

必须同时输出：
1. diagnosis.md：题目理解、子问题拆解、题型判断、数据需求、候选模型比较、初步路线、图表计划、人工确认项和最大风险；
2. diagnosis.json：与 diagnosis.md 语义一致，且符合 schemas/diagnosis.schema.json；
3. 材料清单：题面、附件、模板（如官方提供）的可用性和缺失项。

若当前环境不能直接写文件，分别以 `diagnosis.md` 和 `diagnosis.json` 标题输出两个独立内容块；不得只输出 Markdown。
```

## 阶段推进

以下阶段按 `standard` 执行。`strict` 要求每一 Gate 都完成完整机器核验与证据记录；
`emergency` 只能合并 Gate 2 与 Gate 4 的文档性工作，必须保留 Gate 0、Gate 1、Gate 3 和 Gate 5。

### Gate 0：题目与材料诊断

完成读题、拆题、题型判断、候选模型、数据需求、图表计划、材料可用性检查和人工确认项。必须生成 `diagnosis.md` 与 `diagnosis.json`。

每一问还必须明确现实过程、核心实体、事件与状态变化、题目专属核心矛盾、数据与单位、
确定性/随机性、对子问题递进的作用以及候选机制解释。

### Gate 1：模型路线

AI 记录路线与依据后进入：

```text
采用第 X 条路线，现在进入 Gate 1：模型路线。
```

输出每一问的模型结构、变量、公式、约束、输入输出表和验证方式。

每个子问题必须包含机制链、题目专属洞察、核心矛盾、前后问关系、结构假设、决策空间审计、
降维依据、简化验证、模型有效性测试与图形论证，并区分理论模型、计算模型和实际推荐方案。

### Gate 2：代码计划

模型路线确认后再进入：

```text
模型路线已确认，现在进入 Gate 2：代码计划。
```

输出模块边界、输入文件、输出文件、基线方法、测试方式、随机性控制和失败降级策略；确认后才开始正式实现。

代码计划必须绑定模型变量与代码变量、数学目标与复算函数、硬约束与独立 Validator、随机假设
与 seeds/repeats/interval，以及图表数据来源与论证目的。

### Gate 3：结果确认

完成求解后再进入：

```text
代码和结果已生成，现在进入 Gate 3：结果确认。
```

确认结果表、图表、约束满足性、基线对比、敏感性或误差分析均可追溯。

Gate 3 必须分别判定 `implementation_correctness`、`model_validity` 和 `competition_value`，
不能合并为一个“验证通过”。

### Gate 4：论文确认

结果确认后再进入：

```text
结果已确认，现在进入 Gate 4：论文确认。
```

确认哪些结果可以写入论文，禁止把候选路线、未运行模型或无依据结论写成最终结论。

每一问按“机理→模型→结果→形态解释→决策建议→适用边界”组织；Claim Map 还要绑定结论作用域、
最优性等级、模型有效性证据和现实执行条件。

### Gate 5：最终验收

交稿前按评委视角检查题目理解、模型匹配、数据支撑、约束完整性、结果解释、图表有效性和可复现性。

## 新题禁用训练规则

新题执行流不使用：

- 候选旧题扫描；
- T0-T4 材料等级；
- M1-M5 材料风险；
- 旧题历史晋级计数；
- 旧题测试记录。
