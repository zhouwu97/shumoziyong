# 新题执行流

## 使用场景

比赛当天或模拟赛当天使用。新题执行不是训练闭环，不做 T0-T4、M1-M5 或 stable 判定。

## 运行模式

启动时必须显式选择模式；未指定时使用 `standard`。模式只决定 Gate 的合并方式，不取消路线确认和结果合理性确认。

| 模式 | 适用场景 | 强制确认点 |
|---|---|---|
| `strict` | 训练、赛前模拟、时间充足 | 总控诊断、建模设计、代码计划、结果、论文终审逐阶段确认 |
| `standard` | 普通比赛 | 路线确认、代码执行前确认、结果合理性确认 |
| `emergency` | 比赛后期时间不足 | 模型路线确认、最终结果合理性确认 |

`emergency` 只合并中间 Gate：不得跳过数据来源检查、不得编造结果、不得在模型路线确认前写求解代码，也不得在最终结果合理性确认前写入论文结论。

## 第一阶段入口提示词

```text
你现在执行“新题执行流”，运行模式为 standard。

请读取：
1. docs/workflows/03_新题执行流.md
2. prompt_base/prompt_base_v1.0.md
3. 当前题目最相关的 plugin
4. 当前题目可能相关的 patch
5. checklists/gate_0_problem_diagnosis.md

本轮只做第一阶段：总控诊断。

禁止：
1. 不写完整论文；
2. 不写代码；
3. 不给最终答案；
4. 不直接套高级模型。

输出：
1. 题目理解
2. 子问题拆解
3. 题型判断
4. 数据需求
5. 候选模型比较
6. 初步建模路线
7. 图表计划
8. 人工确认项
9. 最大跑偏风险
```

## 阶段推进

以下阶段描述按 `standard` 执行。`strict` 在每个阶段末暂停；`emergency` 可把总控诊断与建模设计合并为路线确认，把代码、结果分析与写作准备合并到最终结果合理性确认，但必须保留两个强制确认点。

### 1. 总控诊断

只完成读题、拆题、题型判断、候选模型、数据需求、图表计划和人工确认项。

### 2. 建模设计

人工确认路线后再进入：

```text
我确认采用第 X 条路线，现在进入建模设计阶段。
```

输出每一问的模型结构、变量、公式、约束、输入输出表和验证方式。

### 3. 代码和结果

建模路线确认后再进入：

```text
建模路线已确认，现在进入代码和结果阶段。
```

按模块完成数据读取、清洗、基线模型、主模型、结果表、图表和敏感性分析。

### 4. 论文写作

结果确认后再进入：

```text
结果已确认，现在进入论文写作阶段。
```

按问题重述、问题分析、假设、符号、模型建立、求解、结果分析、检验、评价和摘要推进。

### 5. 审稿反查

交稿前按评委视角检查题目理解、模型匹配、数据支撑、约束完整性、结果解释和图表有效性。

## 新题禁用训练规则

新题执行流不使用：

- 候选旧题扫描；
- T0-T4 材料等级；
- M1-M5 材料风险；
- 旧题 stable 计数；
- 旧题测试记录。
