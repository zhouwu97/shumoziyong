# 数模 AI 训练总控工作流 v3

## 1. 系统定位

本系统用于国赛备赛，不是让 AI 自动生成优秀论文，而是训练一套可测试、可复盘、可迭代的数模协作流程。

核心目标：

```text
优秀论文学习
-> 知识库沉淀
-> 提示词分层进化
-> 旧题测试
-> 失败复盘
-> 新题稳定执行
```

执行原则：

1. 新题先总控诊断，不直接写论文。
2. 优化题先建立评价函数或目标函数，不直接上算法。
3. 工程题先拆机制，不直接套模型。
4. 变量多时先降维、分区、规则化或参数化。
5. 结果出来后先解释和检验，再写论文。
6. 审稿未通过，不交最终稿。

## 2. 三条主线

### 2.1 论文学习线

```text
优秀论文 PDF
-> PDF 核验与文本抽取
-> 论文学习卡片
-> 知识卡片 JSON
-> 题型模型库
```

目标是提取可迁移经验，而不是复述论文。

### 2.2 提示词进化线

```text
当前提示词
-> 原提示词问题诊断
-> 论文经验补丁
-> 题型插件
-> 旧题测试
-> 失败复盘
-> 局部修复
```

目标是让提示词越来越能约束 AI 少犯错。

### 2.3 新题执行线

```text
新赛题
-> 总控诊断
-> 人工确认
-> 分模块执行
-> 结果分析
-> 论文写作
-> 审稿修改
```

目标是比赛时稳定产出结构完整、模型合理、结果可信、表达规范的论文。

## 3. 训练闭环总览

完整闭环：

```text
拆优秀论文
-> 生成学习卡片 / JSON
-> 整理题型模型库
-> 用旧题测试
-> 发现 AI 跑偏
-> 失败复盘打标签
-> 局部修改提示词
-> 再测试
-> 形成比赛版工作流
```

第一轮只跑 MVP：

```text

-> 工程优化旧题总控测试
-> 100 分评分
-> 失败复盘
-> 局部改提示词
```

## 5. 阶段 0：工具箱准备

必须准备：

1. 优秀论文原文；
2. 论文学习卡片；
3. 知识卡片 JSON；
4. 基础总控提示词；
5. 题型插件提示词；
6. 论文经验补丁；
7. 旧题测试模板；
8. 失败复盘模板；
9. 训练日志。

## 6. 阶段 1：优秀论文学习

每篇论文按以下顺序处理：

1. PDF 版式核验；
2. 文本抽取；
3. 论文基本信息；
4. 题目理解方式；
5. 子问题拆解；
6. 建模路线；
7. 模型适用性；
8. 数据处理；
9. 图表论证；
10. 结果解释；
11. 模型评价与不足；
12. 可迁移经验；
13. 不可照搬经验；
14. 输出学习卡片和 JSON。

拆解质量评分：

| 项目 | 分值 |
|---|---:|
| 题目理解 | 15 |
| 子问题拆解 | 15 |
| 建模路线 | 20 |
| 模型选择理由 | 15 |
| 数据处理 | 10 |
| 图表论证 | 10 |
| 结果分析 | 10 |
| 可复用经验 | 5 |

低于 80 分必须重拆。

## 7. 阶段 2：题型模型库建设

每累计 3 篇论文更新一次题型模型库。

库类型：

1. 题型库；
2. 模型库；
3. 建模路线库；
4. 数据处理库；
5. 图表论证库；
6. 风险库；
7. 审稿扣分库。

每个条目必须包含：

```text
适用场景
输入
输出
适用条件
常见误用
替代模型
图表建议
审稿风险
```

## 8. 阶段 3：提示词进化

提示词采用三层结构：

```text
基础总控提示词
+ 题型插件提示词
+ 论文经验补丁
```

硬规则：

1. 论文经验默认进入 patch，不直接进入 base。
2. 只有某条规则在 3 个以上不同题型测试中都有效，才允许进入基础总控提示词。
3. 单篇论文产生的特殊技巧不得污染主流程。
4. 每次修改提示词必须记录版本和测试结果。
5. 新版低于旧版时必须回滚。

可进入 base 的规则示例：

- 先总控诊断，不直接写论文；
- 每个模型必须说明为什么适合；
- 数据不足必须说明降级方案；
- 结果必须解释现实意义。

留在 patch 的规则示例：

- A092 的输出功率评价函数；
- 同心圆布局降维；
- 蒙特卡洛截断效率。

## 9. 阶段 4：旧题测试

先跑总控测试，不写论文。

总控诊断 100 分评分表：

| 项目 | 分值 |
|---|---:|
| 题目理解 | 15 |
| 子问题拆解 | 15 |
| 题型判断 | 10 |
| 模型候选与取舍 | 20 |
| 数据需求 | 10 |
| 建模路线 | 15 |
| 图表与结果计划 | 10 |
| 人工确认项 | 5 |

判定规则：

- 80 分以上：进入下一阶段；
- 60-79 分：局部修改提示词后重测；
- 60 分以下：回到题目诊断提示词和模型质疑提示词重构。

总控测试通过后，再跑分模块测试：

1. 解题思路生成；
2. 建模路线设计；
3. 模型选择与质疑；
4. 数据处理与代码实现；
5. 结果分析与图表设计；
6. 论文写作；
7. 摘要优化；
8. 审稿修改。

## 10. 阶段 5：失败复盘

失败标签：

| 标签 | 含义 |
|---|---|
| P1 | 题目理解错误 |
| P2 | 子问题拆解错误 |
| P3 | 模型不匹配 |
| P4 | 高级模型滥用 |
| P5 | 数据需求不清 |
| P6 | 代码不可执行 |
| P7 | 结果只复述数字 |
| P8 | 图表不服务论证 |
| P9 | 摘要空泛 |
| P10 | 审稿没发现关键问题 |

每次失败必须生成复盘卡：

```text
测试题：
使用提示词版本：
失败现象：
失败标签：
原因判断：
对应提示词漏洞：
需要新增的约束：
修改后的提示词片段：
下次测试题型：
是否进入稳定版：
```

每 3 次旧题测试后统计失败标签频率。出现最多的标签决定下一轮优先补强哪个提示词模块。

## 11. 阶段 6：新题执行

正式新题按闸门执行：

```text
总控诊断未通过，不进入模型设计；
模型路线未确认，不写代码；
代码无结果，不写论文；
结果未解释，不写摘要；
审稿未通过，不交最终稿。
```

执行顺序：

1. 总控诊断；
2. 人工确认路线；
3. 建模路线设计；
4. 模型质疑；
5. 数据与代码；
6. 结果图表；
7. 论文初稿；
8. 摘要优化；
9. 审稿修改；
10. 最终检查。

## 12. 阶段 7：国赛 72 小时流程

```text
0-4 小时：读题、拆题、总控路线
4-10 小时：数据检查、模型确认
10-28 小时：代码和初步结果
28-42 小时：结果分析和图表
42-58 小时：论文初稿
58-68 小时：审稿修改
68-72 小时：摘要、排版、最终检查
```

## 13. A092 优化类硬规则

A092 的核心经验：

```text
先建立基础评价模型
-> 再把后续问题转化为约束优化
```

工程优化类题必须执行：

1. 先判断是否需要评价函数或目标函数；
2. 不允许直接上优化算法；
3. 必须说明优化目标、决策变量、约束条件、评价指标；
4. 如果问题包含复杂物理或工程机制，必须先拆解机制；
5. 如果变量维度太高，必须思考如何降维、分区、规则化或参数化；
6. 如果使用遗传算法、粒子群、模拟退火等算法，必须说明为什么不用更简单的遍历、规划或解析方法；
7. 结果必须包含图表、趋势解释、优化前后对比和敏感性分析。

## 14. 赛前冻结机制

赛前 3-5 天冻结主提示词。

冻结后：

1. 不允许重写主流程；
2. 不允许大幅合并提示词；
3. 不允许删除已测试稳定的规则；
4. 只允许新增应急补丁。

应急补丁示例：

```text
预测类应急补丁：
必须先画时间序列趋势图；
必须检查数据量；
必须比较灰色预测、回归、ARIMA；
必须说明误差指标；
不能直接上 LSTM。
```

## 15. MVP 最小验证流程

第一轮只跑：

```text
A092
-> 生成工程优化类提示词补丁
-> 找一道工程优化类旧题
-> 跑总控诊断
-> 按 100 分表评分
-> 生成失败复盘
-> 局部修改提示词
-> 再测一次
```

MVP 通过标准：

1. 总控诊断 >= 80 分；
2. 不出现 P3、P4、P5；
3. 能先建立评价函数或目标函数；
4. 能给出目标、变量、约束、评价指标；
5. 能比较简单方法和智能优化算法；
6. 能提出至少 3 类图表；
7. 能提出敏感性分析方案；
8. 输出中必须包含人工确认项。

## 16. 稳定版判定标准

提示词版本进入 stable 必须满足：

1. 同类旧题测试总控诊断 >= 80 分；
2. 关键失败标签不超过 1 个；
3. 不出现 P3 模型不匹配、P4 高级模型滥用、P5 数据需求不清；
4. 输出中必须包含人工确认项；
5. 至少通过 2 道同类旧题测试后，才能标记为 stable。

## 17. 训练日志规则

每次训练只在 `training_log.md` 记录一行：

```text
日期 | 论文/旧题 | 使用版本 | 测试模块 | 得分 | 失败标签 | 修改内容 | 是否稳定
```

记录目的：

1. 追踪系统进化；
2. 判断哪个提示词版本最稳定；
3. 统计失败标签高频问题；
4. 避免提示词越改越乱。
