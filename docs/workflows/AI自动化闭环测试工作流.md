# AI 自动化闭环测试工作流

## 1. 文档定位

本文件用于指导 AI 代理自动完成数学建模提示词系统的闭环测试链路。

每次执行本文件前，AI 必须先读取 `docs/workflows/AI闭环工作流执行计划模板.md`，生成《本轮闭环执行计划》，再开始候选题扫描和旧题诊断。没有执行计划，本轮闭环无效。

闭环测试不是让 AI 直接解题、写代码或写论文，而是自动完成以下链路：

```text
生成本轮闭环执行计划
-> 扫描候选旧题
-> 本地无 T3 时自动获取官方材料
-> 自动选择最高资格测试题
-> 读取训练材料
-> 判断材料资格和泄漏风险
-> 运行总控诊断
-> 按评分表自评和机评
-> 识别失败标签
-> 生成旧题测试记录
-> 必要时生成失败复盘卡
-> 提出提示词局部修复方案
-> 更新训练日志和版本日志
-> 生成下一轮重测任务
-> 通过材料资格审查后标记候选稳定版
```

核心目标是让 AI 能够在最少人工干预下完成一次可追踪、可复盘、可回滚的提示词测试循环。

自动化的含义是：AI 默认自己完成候选题扫描、材料资格审查、旧题选择、闭环执行、日志更新和下一轮任务生成。人工只负责提供本地没有的官方材料、批准正式提示词修改、批准最终 stable 标记；不得要求人工手动筛题、手动判断泄漏风险或手动拼接测试文件。

## 2. 自动化边界

### 2.1 AI 自动执行

AI 可以自动执行：

1. 扫描项目目录和已有训练材料；
2. 自动扫描候选旧题、附件和题面来源；
3. 自动判断候选题的 T0-T4 等级和 M1-M5 风险；
4. 本地没有 T3 时，自动查找和获取官方题面、附件和模板；
5. 自动校验材料来源、文件完整性和泄漏风险；
6. 自动选择当前可用的最高资格测试题；
7. 读取 base、plugin、patch、学习卡片和知识卡片；
8. 根据旧题材料生成总控诊断；
9. 按 100 分评分表打分并说明扣分原因；
10. 检查 P1-P10 失败标签；
11. 生成旧题测试记录文件；
12. 生成失败复盘卡；
13. 判断应修改 base、plugin 还是 patch；
14. 生成局部提示词补丁建议；
15. 更新 `training_log.md`；
16. 更新 `prompt_versions/prompt_version_log.md`；
17. 生成下一轮重测任务；
18. 汇总当前闭环状态。

### 2.2 必须人工确认

以下节点必须暂停等待人工确认：

1. 是否允许把 AI 生成的提示词补丁写入正式文件；
2. 是否允许将某个版本标记为 `stable`；
3. 是否允许把单篇论文经验提升到 `prompt_base`；
4. 是否进入论文写作、代码实现或正式新题执行阶段；
5. AI 已完成本地扫描和官方材料自动获取尝试，但仍无法取得官方题面、附件或输出模板时，由人工补材料。

以下事项不需要人工确认，AI 必须自动处理：

1. 判断某轮测试是 T0、T1、T2、T3 还是 T4；
2. 判断是否存在 M1-M5 材料风险；
3. 判断 T0/T1/T2 不计入 stable；
4. 判断同题、同附件或同结果来源只能作为流程测试记录；
5. 在已有候选题中选择更适合作为 T3 的旧题。

### 2.3 禁止自动执行

闭环测试阶段禁止 AI 自动执行：

1. 不得直接写完整竞赛论文；
2. 不得直接写求解代码；
3. 不得跳过总控诊断进入模型求解；
4. 不得因为一次旧题通过就标记 `stable`；
5. 不得把单篇论文特殊技巧直接写入 `prompt_base`；
6. 不得大幅重写已有提示词，只允许局部修复；
7. 不得删除旧版本、旧测试记录或失败复盘。
8. 不得把优秀论文 PDF 当作无泄漏官方题面计入 `stable`；
9. 不得把同题论文经验 patch 测同一道题的结果计入泛化通过次数。

```text

```

## 5. S0 项目预检

### 5.1 输入

项目根目录。

### 5.2 自动检查项

AI 必须检查以下文件是否存在：

```text
docs/workflows/数模AI训练总控工作流v3.md
training_log.md
prompt_base/prompt_base_v1.0.md
prompt_plugins/plugin_optimization_v1.md
tests/old_problems/旧题测试_工程优化_模板.md
reviews/failure_cards/失败复盘_模板.md
prompt_versions/prompt_version_log.md
```

### 5.3 输出

预检摘要：

```text
项目预检结果：
- 必需文件是否齐全：
- 缺失文件：
- 当前可用 base：
- 当前可用 plugin：
- 当前可用 patch：
- 当前已有学习卡片：
- 当前已有知识卡片：
- 当前稳定版本：
- 是否允许进入闭环测试：
```

### 5.4 失败分支

如果缺少关键文件：

1. 缺少模板：AI 先根据现有总控文档生成模板草案；
2. 缺少 base：停止闭环，要求先建立 base；
3. 缺少 plugin：只运行 base 测试，并标记插件缺失风险；
4. 缺少 patch：只运行 base + plugin 测试；
5. 缺少旧题：生成旧题材料需求清单，等待人工提供。

## 6. S1 材料装载

### 6.1 输入材料

每次闭环测试至少需要：

```text
base 提示词
plugin 提示词
patch 提示词
论文学习卡片
知识卡片 JSON
旧题原文
旧题附件说明
```

### 6.2 装载顺序

AI 必须按以下顺序读取材料：

1. `docs/workflows/数模AI训练总控工作流v3.md`
2. `training_log.md`
3. `prompt_base/prompt_base_v1.0.md`
4. 对应题型的 plugin
5. 对应论文经验 patch
6. 对应论文学习卡片
7. 对应知识卡片 JSON
8. 旧题原文和附件说明
9. 旧题测试模板
10. 失败复盘模板

### 6.3 材料摘要格式

AI 装载完成后，必须输出：

```text
本轮材料摘要：
- 题型：
- 旧题：
- base：
- plugin：
- patch：
- 学习卡片：
- 知识卡片：
- 旧题附件：
- 本轮测试目标：
- 本轮禁止事项：
```

### 6.4 材料资格与泄漏风险分级

每轮测试必须先判断材料资格。评分通过不等于可计入 stable，只有材料资格通过的测试才允许进入稳定性计数。

| 等级 | 名称 | 材料来源 | 是否可跑闭环 | 是否可计入 stable |
|---|---|---|---|---|
| T0 | 流程冒烟测试 | 优秀论文 PDF、论文摘要、论文问题分析、论文结果反推题意 | 是 | 否 |
| T1 | 材料受限测试 | 官方题面不完整、缺附件、缺结果模板、允许少量人工补题意 | 是 | 否 |
| T2 | 官方题面测试 | 独立官方题面和附件齐全，但测试题与训练论文同题或同附件 | 是 | 否 |
| T3 | 泛化测试 | 独立官方题面和附件齐全，且不与当前 patch 来源论文同题 | 是 | 是 |
| T4 | 稳定候选测试 | 至少两道 T3 同题型旧题均通过，且无 P3/P4/P5 | 是 | 可建议 stable |

### 6.5 附加材料风险标签

除 P1-P10 外，AI 必须检查以下材料风险标签：

| 标签 | 含义 | 处理方式 |
|---|---|---|
| M1 | 答案泄漏风险：材料来自优秀论文、解析、参考答案或含结果的复述 | 可跑流程，不计 stable |
| M2 | 官方附件或结果模板缺失 | 可跑材料受限测试，不计 stable |
| M3 | 同题污染：测试题与 patch 来源论文为同一赛题、同一附件或同一结果格式 | 可跑同题验证，不计 stable |
| M4 | 经验照搬风险：输出直接复用了论文特殊布局、公式、数值或结论 | 若影响诊断质量，生成失败复盘 |
| M5 | 人工补题意过多：关键约束来自人工总结而非官方题面 | 可跑流程，不计 stable |

### 6.6 材料资格摘要

AI 必须在闭环摘要和旧题测试记录中写入：

```text
材料资格审查：
- 测试等级：T0 / T1 / T2 / T3 / T4
- 是否存在 M1：
- 是否存在 M2：
- 是否存在 M3：
- 是否存在 M4：
- 是否存在 M5：
- 是否允许跑闭环：
- 是否允许计入 stable：
- 不计入 stable 的原因：
```

### 6.7 候选旧题自动筛选

如果用户没有指定旧题，AI 必须自动从项目内候选材料中筛选，不得停止要求人工手动选题。

候选材料扫描范围：

```text
tests/old_problems/
高教社杯全国大学生数学建模竞赛优秀论文/
papers/
output/
tmp/
项目根目录下的 PDF、DOCX、XLSX、CSV、TXT、MD
```

自动筛选步骤：

1. 列出所有可能的题面文件、附件文件和结果模板；
2. 提取年份、题号、题型关键词和材料来源；
3. 判断是否官方题面、优秀论文、学习卡片、解析、答案或结果模板；
4. 判断是否与当前 patch 来源论文同题；
5. 判断附件是否完整；
6. 给每个候选题打 T0-T4 等级和 M1-M5 标签；
7. 优先选择最高等级候选题；
8. 如果目标是 stable，优先选择 T3；没有 T3 时不得用 T0/T1/T2 顶替；
9. 生成候选题排序表和选择理由；
10. 如果没有可用 T3，自动生成“材料缺口清单”和“下一步材料获取任务”。

候选题排序表格式：

```text
| 排名 | 候选题 | 材料路径 | 附件状态 | 题型判断 | T等级 | M标签 | 是否可跑 | 是否可计入stable | 选择/排除理由 |
|---:|---|---|---|---|---|---|---|---|---|
```

自动选择规则：

```text
如果存在 T3：
  选择综合得分最高的 T3 作为本轮正式泛化测试。

如果不存在 T3，但存在 T0/T1/T2：
  先自动尝试获取官方题面、附件和模板；
  只有官方材料获取失败后，才选择最适合的 T0/T1/T2 只跑流程冒烟或材料受限测试，同时生成 T3 材料缺口清单。

如果没有任何可跑材料：
  生成材料获取任务，不生成虚假的测试记录，不编造题面或附件。
```

T3 候选综合得分：

| 指标 | 分值 |
|---|---:|
| 官方题面完整 | 25 |
| 官方附件完整或题目不需要附件 | 20 |
| 非训练论文同题 | 20 |
| 工程优化/布局优化/资源配置匹配度 | 15 |
| 有明确输出要求或模板 | 10 |
| 不含优秀论文/答案/解析泄漏 | 10 |

低于 80 分的候选题不得作为 T3。

### 6.8 官方材料自动获取

如果本地扫描后没有 T3/T4 候选，AI 必须先尝试自动获取官方材料，不得直接把任务交还给人工。

自动获取顺序：

1. 优先使用本地已有官方题面、附件、压缩包或模板；
2. 若本地没有，使用网络搜索官方来源；
3. 优先访问竞赛主办方、承办方、学校镜像或官方发布页；
4. 只下载官方题面、官方附件、官方结果模板或官方数据；
5. 不下载、不读取获奖论文、题解、参考答案、解析文章、论坛答案或含结果复述；
6. 下载失败时可尝试国内镜像或可信学校站点，但必须标注来源；
7. 下载后必须保存来源 URL、文件名、时间和用途。

允许自动获取的材料类型：

```text
官方赛题 PDF / DOC / DOCX
官方附件 ZIP / XLSX / CSV / TXT / MAT
官方结果模板 XLSX
官方数据说明
```

禁止作为 T3 输入的材料：

```text
优秀论文 PDF
获奖论文
参考答案
题解解析
博客复述
论坛讨论
含最终数值结果的说明
```

材料保存目录：

```text
official_materials/{年份}_{题号}/
  problem/
  attachments/
  templates/
  manifest.md
```

如果目录不存在，AI 应自动创建。`manifest.md` 必须记录：

```text
# 官方材料清单_{年份}_{题号}

## 来源

| 文件 | 来源 URL 或本地来源 | 获取时间 | 用途 | 是否官方 | 备注 |
|---|---|---|---|---|---|

## 完整性检查

- 是否有官方题面：
- 是否有附件：
- 是否有结果模板：
- 是否包含答案泄漏：
- 是否可作为 T3：

## 风险判断

- M1：
- M2：
- M3：
- M4：
- M5：
```

自动获取后的判定：

```text
如果官方题面 + 必要附件 + 输出要求齐全，且无 M1/M3/M5：
  将候选题升级为 T3，进入正式泛化测试。

如果只有官方题面但附件缺失：
  记为 T1/T2，不计 stable，继续尝试获取附件。

如果只找到优秀论文或解析：
  只能记为 T0，不计 stable。

如果网络失败或来源无法确认官方性：
  生成官方材料获取失败报告，不编造材料。
```

材料获取失败报告位置：

```text
output/closed_loop/官方材料获取失败_{题型}_{日期}.md
```

失败报告必须包含：

```text
尝试的关键词
尝试的来源
失败原因
已找到但不能用作 T3 的材料
仍缺失的材料
需要人工提供的最小清单
提供后从哪一步继续
```

## 7. S2 旧题任务建立

### 7.1 旧题选择规则

工程优化类旧题优先选择具备以下特征的题：

1. 存在最大化或最小化目标；
2. 存在工程、物理、空间、资源或机制约束；
3. 需要定义目标函数、评价函数或综合指标；
4. 决策变量和约束较多；
5. 需要图表解释结果；
6. 可以检验 AI 是否滥用高级模型。

### 7.2 stable 计数旧题选择规则

如果本轮目标是验证稳定性或计入 stable，通过旧题必须同时满足：

1. 使用官方独立题面；
2. 官方附件和输出模板齐全，或题目本身不需要附件；
3. 旧题不来自已学习论文的同一题号；
4. 旧题不与当前 patch 来源论文共享关键结果、附件或标准答案；
5. 不读取优秀论文全文、参考答案、获奖论文结果或解析；
6. 只允许读取抽象化的 base、plugin、patch 规则，不允许读取同题具体数值和布局结论；
7. 如果无法确认是否同题污染，默认不计入 stable。

### 7.3 任务卡格式

AI 必须先生成任务卡：

```text
# 闭环测试任务卡

- 测试日期：
- 旧题来源：用户指定 / AI 自动筛选
- 旧题年份与题号：
- 题型：
- 候选题排序表路径：
- 本轮选择理由：
- 使用 base：
- 使用 plugin：
- 使用 patch：
- 参考学习卡片：
- 参考知识卡片：
- 材料资格等级：
- 材料风险标签：
- 是否允许计入 stable：
- 测试模块：总控诊断
- 是否写论文：否
- 是否写代码：否
- 是否直接给最终答案：否
- 本轮重点检查：
  1. 是否先建立评价函数或目标函数；
  2. 是否明确目标、变量、约束、评价指标；
  3. 是否比较简单方法和高级优化算法；
  4. 是否给出数据需求；
  5. 是否给出图表计划；
  6. 是否给出人工确认项。
```

## 8. S3 总控诊断生成

### 8.1 执行原则

AI 只生成总控诊断，不写论文，不写代码，不直接给最终解。

### 8.2 必须输出的结构

总控诊断必须包含：

```text
一、训练材料理解
二、材料资格审查
三、旧题总控诊断
四、优化类专项检查
五、模型候选与模型质疑
六、数据需求与代码准备
七、图表与结果计划
八、人工确认项
九、100 分自评
十、失败标签检查
十一、最终判定
```

### 8.3 工程优化硬规则

如果旧题属于工程优化类，必须检查：

1. 是否先建立评价函数或目标函数；
2. 是否先拆工程机制；
3. 是否明确优化对象；
4. 是否明确优化目标；
5. 是否明确决策变量；
6. 是否明确约束条件；
7. 是否明确评价指标；
8. 是否比较简单方法与智能优化算法；
9. 是否考虑降维、分区、规则化或参数化；
10. 是否规划优化前后对比和敏感性分析。

### 8.4 自动化输出位置

总控诊断全文应写入闭环摘要文件：

```text
output/closed_loop/闭环摘要_{题型}_{年份题号}_{日期}.md
```

## 9. S4 评分与失败标签检测

### 9.1 100 分评分表

AI 必须按以下评分表打分：

| 项目 | 分值 | 自动评分依据 |
|---|---:|---|
| 题目理解 | 15 | 是否抓住背景、核心矛盾、已知未知条件 |
| 子问题拆解 | 15 | 是否拆清每一问的表面任务和数学任务 |
| 题型判断 | 10 | 是否正确识别评价、预测、优化、仿真等题型 |
| 模型候选与取舍 | 20 | 是否给出候选模型、适用原因、局限和替代方案 |
| 数据需求 | 10 | 是否列出数据项、来源、用途、缺失处理和风险 |
| 建模路线 | 15 | 是否形成输入、处理、输出链路 |
| 图表与结果计划 | 10 | 图表是否服务结论，是否覆盖对比和敏感性分析 |
| 人工确认项 | 5 | 是否列出进入下一阶段前必须确认的问题 |

### 9.2 关键失败标签

AI 必须检查以下标签：

| 标签 | 含义 | 关键程度 |
|---|---|---|
| P1 | 题目理解错误 | 高 |
| P2 | 子问题拆解错误 | 高 |
| P3 | 模型不匹配 | 最高 |
| P4 | 高级模型滥用 | 最高 |
| P5 | 数据需求不清 | 最高 |
| P6 | 代码不可执行 | 中，本阶段通常不触发 |
| P7 | 结果只复述数字 | 中 |
| P8 | 图表不服务论证 | 中 |
| P9 | 摘要空泛 | 低，本阶段通常不触发 |
| P10 | 审稿没发现关键问题 | 中 |

### 9.3 材料风险标签

材料风险标签不一定代表提示词失败，但会影响闸门判定和 stable 资格。

| 标签 | 是否算 P 类失败 | 是否影响 stable | 说明 |
|---|---|---|---|
| M1 | 否 | 是 | 答案泄漏风险，不能计入 stable |
| M2 | 否 | 是 | 附件或结果模板缺失，不能进入正式求解 |
| M3 | 否 | 是 | 同题污染，不能证明泛化能力 |
| M4 | 视情况 | 是 | 若出现照搬，应按 P3/P4 或 P10 复盘 |
| M5 | 否 | 是 | 人工补题意过多，不能作为独立测试 |

### 9.4 自动判定规则

```text
总分 >= 80 且无 P3/P4/P5 且材料等级为 T3/T4：
  通过正式总控诊断，可以计入 stable 候选统计。

总分 >= 80 且无 P3/P4/P5 但材料等级为 T0/T1/T2：
  通过流程或材料受限诊断，但不能计入 stable。

总分 >= 80 但出现 P3/P4/P5：
  不通过，必须失败复盘。

总分 60-79：
  不通过，局部修改提示词后重测。

总分 < 60：
  不通过，重构总控诊断提示词或题型插件。
```

## 13. S8 提示词局部修复建议

### 13.1 修复层级判断

AI 必须按以下规则判断修改层级：

| 问题类型 | 修改层级 |
|---|---|
| 所有题型都应遵守的基本闸门缺失 | base |
| 某一题型通用规则缺失 | plugin |
| 单篇论文经验迁移不到位 | patch |
| 只是本次输出执行不到位 | 不改提示词，重测 |
| 旧题材料不完整 | 不改提示词，补材料 |

### 13.2 base 修改限制

只有满足以下条件之一，才允许建议修改 base：

1. 同一问题在 3 个以上不同题型测试中反复出现；
2. 缺失的是通用工作流闸门；
3. 缺失的是通用人工确认项；
4. 缺失的是通用数据降级规则；
5. 人工明确要求提升到 base。

### 13.3 plugin 修改条件

如果问题只属于某一题型，例如优化类，则优先修改 plugin：

1. 优化目标不清；
2. 决策变量不清；
3. 约束条件不清；
4. 高级优化算法滥用；
5. 缺少简单方法优先检查；
6. 缺少敏感性分析。

### 13.4 patch 修改条件

如果问题只与某篇论文经验有关，则修改 patch：

1. A092 的评价函数先行经验没有被触发；
2. A127 的分区、分层、规则化经验没有被触发；
3. AI 照搬了论文中的特殊公式、布局或数值；
4. AI 没有说明哪些经验不可迁移。

### 13.5 修复草案格式

AI 生成修复草案时必须使用以下格式：

```text
# 提示词修复草案_{题型}_{年份题号}_{日期}

## 1. 触发原因

## 2. 对应失败标签

## 3. 建议修改层级

## 4. 建议修改文件

## 5. 原提示词漏洞

## 6. 建议新增片段

待插入的提示词片段：

    在这里写入建议新增或替换的提示词片段。

## 7. 不建议修改的内容

## 8. 重测标准
```

注意：修复草案默认写入 `output/closed_loop/`，不得直接覆盖正式提示词文件。只有人工确认后，才允许写入 `prompt_base/`、`prompt_plugins/` 或 `prompt_patches/`。

## 14. S9 日志与版本记录更新

### 14.1 更新 training_log.md

每次闭环测试必须在 `training_log.md` 增加一行：

```text
| 日期 | 论文/旧题 | 使用版本 | 测试模块 | 得分 | 失败标签 | 修改内容 | 是否稳定 |
```

```text
| 2026-07-02 | 2021-B 工程优化旧题 | prompt_base_v1.0 + plugin_optimization_v1 + patch_A092_engineering_optimization | 总控诊断 | 76 | P5 | T3，生成失败复盘，建议补强数据需求表 | 否 |
```


### 14.3 更新稳定版本记录

只有当同类旧题至少通过 2 次 T3 或 T4 测试，且没有 P3/P4/P5/M1/M2/M3/M5，才允许建议更新稳定版本记录。

### 14.4 更新 prompt_version_log.md

如果人工确认了正式提示词修改，AI 必须新增版本记录：

```text
| 日期 | 版本 | 层级 | 状态 | 来源论文/旧题 | 引入经验 | 修改模块 | 测试题 | 得分 | 失败标签 | 是否稳定 | 备注 |
```

状态规则：

```text
draft：刚生成，未测试
testing：已进入旧题测试
stable：至少通过 2 道同类不同题号的 T3/T4 官方旧题
rollback：已回滚
frozen：赛前冻结
```

## 15. S10 重测任务生成

### 15.1 触发条件

以下情况必须生成重测任务：

1. 修改了任何提示词；
2. 总分低于 80；
3. 出现 P3/P4/P5；
4. 旧题材料补充后需要重跑；
5. AI 输出执行不到位但提示词不需要修改。
6. 材料等级为 T0/T1/T2，但用户希望验证 stable；
7. 出现 M1/M2/M3/M5，导致本轮不能计入 stable；
8. 出现 M4，需要更换无同题泄漏材料验证是否仍会照搬。

### 15.2 重测任务格式

```text
# 重测任务_{题型}_{年份题号}_{日期}

## 1. 重测原因

## 2. 上轮失败标签

## 3. 本轮使用版本

## 4. 本轮重点检查

## 5. 必须改善项

## 6. 通过标准

## 7. 如果仍失败
```

### 15.3 重测通过标准

工程优化类重测必须满足：

1. 总控诊断得分 >= 80；
2. 不出现 P3/P4/P5；
3. 明确评价函数或目标函数；
4. 明确优化目标、决策变量、约束条件和评价指标；
5. 比较简单方法和智能优化算法；
6. 有数据需求表；
7. 有图表计划；
8. 有敏感性分析计划；
9. 有人工确认项。
10. 如果要计入 stable，材料等级必须为 T3 或 T4；
11. 如果要计入 stable，不得出现 M1/M2/M3/M5；
12. 不得读取同题优秀论文、参考答案或含结果解析。

### 15.4 重测任务类型

AI 生成重测任务时必须说明重测类型：

| 类型 | 适用情况 | 目标 |
|---|---|---|
| 材料补齐重测 | 缺官方题面、附件或模板 | 从 T0/T1 提升到 T2 |
| 换题泛化重测 | 同题污染或答案泄漏 | 从 T0/T1/T2 提升到 T3 |
| 提示词修复重测 | 出现 P3/P4/P5 或 M4 照搬 | 验证修复是否有效 |
| 稳定候选重测 | 已有 1 道 T3 通过 | 补足第 2 道独立 T3 测试 |

## 16. S11 稳定性候选判断

### 16.1 候选稳定版条件

同一题型满足以下条件时，AI 可以建议标记为候选稳定版：

1. 至少通过 2 道同类但不同题号的 T3 官方旧题；
2. 两次总控诊断均 >= 80；
3. 两次均未出现 P3/P4/P5；
4. 输出结构稳定；
5. 人工确认项完整；
6. 数据需求清楚；
7. 模型选择没有滥用高级算法；
8. 图表计划能支撑结论。
9. 两次均未出现 M1/M2/M3/M5；
10. 两次测试题均不与当前 patch 来源论文同题；
11. 两次测试不能使用优秀论文全文、参考答案或结果解析作为题面；
12. 至少一次测试应使用与训练论文不同背景的同类工程优化题，验证迁移能力。

### 16.2 不允许稳定的情况

出现以下任一情况，不得标记 `stable`：

1. 只通过 1 道旧题；
2. 通过但出现 P3/P4/P5；
3. 通过依赖大量人工补救；
4. 测试题型与目标题型不一致；
5. 只是 patch 有效，base 或 plugin 仍不稳定；
6. 新版低于旧版；
7. 缺少测试记录或失败复盘。
8. 旧题来自优秀论文 PDF 或参考答案；
9. 旧题与 patch 来源论文同题、同附件或同结果模板；
10. 材料风险存在 M1/M2/M3/M5；
11. 两道通过测试只是同一道题的不同优秀论文样本。

### 16.3 通过次数口径

训练日志中的“通过旧题数量”只统计 T3/T4 测试。

以下测试必须单独备注，但不得增加通过次数：

1. T0 流程冒烟测试；
2. T1 材料受限测试；
3. T2 同题官方题面测试；
4. 使用优秀论文 PDF 作为题面来源的测试；
5. 使用同题论文 patch 测同一道官方题的测试；
6. 人工补齐关键约束的测试。

```text
   - 不得编造官方材料。

5. 先做材料资格审查，明确：
   - 测试等级是 T0、T1、T2、T3 还是 T4；
   - 是否存在 M1 答案泄漏风险；
   - 是否存在 M2 附件或结果模板缺失；
   - 是否存在 M3 同题污染；
   - 是否存在 M4 经验照搬风险；
   - 是否存在 M5 人工补题意过多；
   - 是否允许跑闭环；
   - 是否允许计入 stable。

6. 生成闭环测试任务卡，明确：
   - 旧题来源是用户指定还是 AI 自动筛选；
   - 旧题年份与题号；
   - 题型；
   - 候选题排序表路径；
   - 本轮选择理由；
   - 使用版本；
   - 材料资格等级；
   - 材料风险标签；
   - 是否允许计入 stable；
   - 测试模块；
   - 禁止写论文；
   - 禁止写代码；
   - 禁止直接给最终答案。

7. 运行旧题总控诊断，必须输出：
   - 训练材料理解；
   - 材料资格审查；
   - 旧题总控诊断；
   - 优化类专项检查；
   - 模型候选与模型质疑；
   - 数据需求与代码准备；
   - 图表与结果计划；
   - 人工确认项；
   - 100 分自评；
   - 失败标签检查；
   - 最终判定。

8. 按 100 分评分表打分，并检查 P1-P10 失败标签和 M1-M5 材料风险标签。

9. 根据闸门规则判断：
   - 总分是否 >= 80；
   - 是否出现 P3/P4/P5；
   - 材料等级是否为 T3/T4；
   - 是否出现 M1/M2/M3/M5；
   - 是否允许进入下一阶段；
   - 是否允许计入 stable；
   - 是否必须失败复盘；
   - 是否需要修改提示词。

10. 自动生成文件：
   - tests/old_problems/旧题测试_{题型}_{年份题号}_{日期}.md
   - output/closed_loop/闭环摘要_{题型}_{年份题号}_{日期}.md
   - output/closed_loop/候选题排序_{题型}_{日期}.md
   - official_materials/{年份}_{题号}/manifest.md，如果自动获取官方材料
   - output/closed_loop/官方材料获取失败_{题型}_{日期}.md，如果获取失败
   - 如果失败，生成 reviews/failure_cards/失败复盘_{题型}_{年份题号}_{日期}.md
   - 如果需要修改提示词，生成 output/closed_loop/提示词修复草案_{题型}_{年份题号}_{日期}.md
   - 如果需要重测，生成 output/closed_loop/重测任务_{题型}_{年份题号}_{日期}.md

11. 更新 training_log.md：
   - 增加本次测试记录；
   - 更新失败标签统计；
   - 写明材料等级和 M 标签；
   - 只有 T3/T4 通过才允许更新稳定版本记录。

12. 如果人工已经明确批准正式修改提示词，则更新对应提示词文件和 prompt_versions/prompt_version_log.md。
   如果没有人工批准，只能生成修复草案，不得覆盖正式提示词。

13. 最后输出闭环报告：
   - 本轮是否通过；
   - 得分；
   - 失败标签；
   - 材料等级；
   - 材料风险标签；
   - 是否计入 stable；
   - 是否由 AI 自动选题；
   - 是否尝试自动获取官方材料；
   - 候选题排序表路径；
   - 生成了哪些文件；
   - 哪个提示词层级需要修；
   - 下一轮怎么重测；
   - 是否接近 stable。

硬规则：
- 不写论文；
- 不写代码；
- 不直接给最终答案；
- 不跳过总控诊断；
- 不把单篇论文经验直接写入 base；
- 不因一次通过标记 stable；
- 不把 T0/T1/T2 计入 stable；
- 不把同题优秀论文样本计入 stable；
- 不用优秀论文 PDF 代替无泄漏官方题面；
- 本地无 T3 时必须先尝试自动获取官方材料；
- 官方材料获取失败才允许停下来要人工补材料；
- 不删除旧记录；
- 所有失败必须可复盘；
- 所有修改必须可回滚。
```

## 18. 自动化执行状态机

```text
INIT
  -> PREFLIGHT
  -> LOAD_MATERIALS
  -> LOCAL_CANDIDATE_SCAN
  -> OFFICIAL_MATERIAL_ACQUISITION
  -> MATERIAL_QUALIFICATION
  -> BUILD_TASK
  -> RUN_DIAGNOSIS
  -> SCORE
  -> GATE

GATE:
  if score >= 80 and no P3/P4/P5 and material is T3/T4 and no M1/M2/M3/M5:
      -> WRITE_TEST_RECORD
      -> UPDATE_LOG
      -> CHECK_STABLE_CANDIDATE
      -> DONE

  if score >= 80 and no P3/P4/P5 but material is T0/T1/T2 or has M1/M2/M3/M5:
      -> WRITE_TEST_RECORD
      -> UPDATE_LOG
      -> WRITE_RETEST_TASK
      -> DONE_AS_LIMITED_TEST

  if score < 80 or has P3/P4/P5:
      -> WRITE_TEST_RECORD
      -> WRITE_FAILURE_CARD
      -> WRITE_FIX_DRAFT
      -> UPDATE_LOG
      -> WRITE_RETEST_TASK
      -> WAIT_HUMAN_CONFIRMATION

WAIT_HUMAN_CONFIRMATION:
  if fix approved:
      -> APPLY_FIX
      -> UPDATE_VERSION_LOG
      -> RETEST

  if fix rejected:
      -> WRITE_RETEST_TASK_WITH_NO_PROMPT_CHANGE
      -> RETEST

RETEST:
  -> LOAD_MATERIALS
  -> RUN_DIAGNOSIS
```

## 20. 第一轮推荐执行方案

当前项目建议分两轮执行，不要把流程测试和稳定性测试混在一起。

### 20.1 第一轮：流程冒烟测试

第一轮用于验证自动化链路能跑通，可以使用已经学习过的定日镜场材料，但必须标记为 T0/T1/T2，不计入 stable。

```text
题型：工程优化
base：prompt_base_v1.0
plugin：plugin_optimization_v1
patch：patch_A092_engineering_optimization
参考论文：2023_A092
测试模块：旧题总控诊断
目标：验证 AI 是否能先建立评价函数或目标函数，再进入优化模型设计
测试等级：T0/T1/T2
是否计入 stable：否
```

如果旧题偏空间布局、阵列规划、遮挡、光伏、光热或工程布置，则可叠加：

```text
patch_A127_engineering_layout_optimization
```

但第一次 MVP 建议只选一个主 patch，避免无法判断是哪条经验生效。

### 20.2 第二轮：正式泛化测试

第二轮用于验证工程优化提示词是否真正能迁移，必须选择非 2023-A、非定日镜场、非同题优秀论文来源的官方旧题。

```text
题型：工程优化 / 布局优化 / 资源配置 / 调度优化
base：prompt_base_v1.0
plugin：plugin_optimization_v1
patch：仅使用抽象化工程优化 patch，不读取同题具体结果
测试材料：独立官方题面 + 官方附件 + 输出要求
测试等级：T3
是否计入 stable：满足得分和标签条件后可以计入
```

正式泛化测试的选题优先级：

1. 与训练论文不同工程背景；
2. 仍然需要评价函数或目标函数；
3. 有明确决策变量和约束；
4. 有附件或数据处理要求；
5. 能检验是否滥用遗传算法、粒子群、神经网络等高级模型。

### 20.3 第三轮：稳定候选测试

如果已有 1 道 T3 工程优化旧题通过，再选择第 2 道不同题号的 T3 旧题。

只有两道不同 T3 旧题都通过，且没有 P3/P4/P5/M1/M2/M3/M5，才允许建议工程优化进入 stable 候选。

## 21. 本工作流的完成标准

一次自动化闭环完成，必须产出：

1. 旧题测试记录；
2. 闭环摘要；
3. 训练日志更新；
4. 闸门判定；
5. 如失败，必须有失败复盘；
6. 如需修改，必须有修复草案；
7. 如需重测，必须有重测任务；
8. 最终说明是否可以进入下一阶段。

当同一题型至少完成 2 次 T3/T4 通过闭环后，才允许进入稳定版评估。
