# 经验升级规则

本规则用于决定论文学习得到的经验是否可以从单篇 patch 升级到 plugin 或 base。它只用于旧题闭环后的规则治理，不用于新题执行中的临场发挥。

## 1. 分层原则

```text
单篇论文经验
-> patch candidate
-> 旧题闭环验证
-> plugin candidate
-> 多题稳定验证
-> plugin 正式规则
-> 跨题型验证
-> base candidate
-> base 正式规则
```

默认规则：

1. 单篇论文经验只能进入 `prompt_patches/`，状态为 `candidate`。
2. 未经过旧题闭环的经验不得进入 runtime 默认导出包。
3. 单篇论文中的具体公式、参数、背景、数据结论和代码结构不得升级。
4. 升级只升级“可迁移方法”，不升级“该论文答案”。

## 2. patch candidate

满足以下条件时，可以建立 patch candidate：

1. 已生成学习卡片、知识卡片 JSON 和 patch 草案；
2. patch 写明适用条件、不适用条件和不可照搬清单；
3. patch 不修改 base/plugin；
4. `prompt_patches/patch_index.json` 已登记；
5. `training_log.md` 已记录本次论文学习事实。

## 3. verified candidate

满足以下条件时，可以把 patch 从 `candidate` 调整为 `verified_candidate`：

1. 至少通过 2 道 T3/T4 官方旧题的总控诊断闭环；
2. 至少包含 1 道相似旧题和 1 道不相似旧题；
3. 总控诊断评分均不低于 80；
4. 未出现 P3/P4/P5；
5. 未出现 M1/M2/M3/M5；
6. 测试题不与 patch 来源论文同题、同附件或共享关键结论；
7. 测试记录、闭环摘要和训练日志完整。

`verified_candidate` 可以进入对应 runtime profile 的默认导出包，但仍不等于 stable。

## 4. plugin candidate

同一类题中至少 3 篇不同论文、不同题号反复出现同一经验，并且该经验通过旧题闭环验证后，可以提出 plugin candidate。

例子：

```text
先建立评价函数，再转化为优化问题
```

这类经验可能适合工程优化 plugin，因为它不依赖单篇论文背景。

不能升级的例子：

```text
同心圆布局降维
```

它只适合特定空间布局背景，应留在 A092/A127 patch 中。

## 5. plugin 正式规则

plugin candidate 满足以下条件后，才允许建议写入正式 plugin：

1. 通过至少 3 道 T3/T4 官方旧题；
2. 覆盖至少 2 个不同背景；
3. 对比旧版本没有引入新的 P3/P4/P5；
4. 有失败复盘证明该规则解决的是稳定问题，而不是单题偶然问题；
5. 人工确认升级范围。

## 6. base candidate

只有跨题型也成立的经验，才允许提出 base candidate。

可考虑进入 base 的经验通常是：

1. 先诊断，后建模；
2. 先写输入、输出、变量、指标和约束；
3. 先解释模型适用性，再使用模型；
4. 没有人工确认，不进入下一阶段。

不得进入 base 的经验通常是：

1. 某个题型特有的模型路线；
2. 某篇论文的具体公式；
3. 某种算法偏好；
4. 某个背景下的参数设置；
5. 只在一个旧题上表现好的表达模板。

## 7. 降级和回滚

出现以下情况时，应降级或回滚：

1. 新 patch 导致不相似旧题乱套；
2. 旧题闭环出现 P3/P4/P5；
3. AI 把不可照搬内容迁移到新题；
4. runtime_pack 变长但诊断质量没有提升；
5. 新版本表现低于旧版本。

降级顺序：

```text
stable/plugin 正式规则
-> plugin candidate
-> verified_candidate patch
-> candidate patch
-> archived
```

任何降级都必须写入 `training_log.md` 和对应失败复盘。
