# 经验升级规则

本规则用于决定论文学习得到的经验是否可以从单篇 patch 升级到 plugin 或 base。它只用于旧题闭环后的规则治理，不用于新题执行中的临场发挥。

## 1. 分层原则

```text
单篇论文经验
-> draft
-> review_ready
-> regression_verified
-> competition_evidenced
-> deprecated
```

默认规则：

1. 单篇论文经验只能进入 `prompt_patches/`，完成结构审查后状态为 `review_ready`。
2. 未经过旧题闭环的经验不得进入 runtime 默认导出包。
3. 单篇论文中的具体公式、参数、背景、数据结论和代码结构不得升级。
4. 升级只升级“可迁移方法”，不升级“该论文答案”。

## 2. review_ready

满足以下条件时，可以进入 `review_ready`：

1. 已生成学习卡片、知识卡片 JSON 和 patch 草案；
2. patch 写明适用条件、不适用条件和不可照搬清单；
3. patch 不修改 base/plugin；
4. `prompt_patches/patch_index.json` 已登记；
5. `training_log.md` 已记录本次论文学习事实。

## 3. regression_verified

只有正向、边界和负控三类对照都具有 baseline、treatment、comparison review 和双方 Evidence Manifest，且 Policy 现场派生结果均为 pass，才可进入 `regression_verified`。

1. 至少覆盖 3 道独立题目和 2 个年份；
2. 正向、边界和负控使用独立可复核的完整 Gate 0-5 运行；
3. 总控诊断评分均不低于 80；
4. 未出现 P3/P4/P5；
5. 未出现 M1/M2/M3/M5；
6. 测试题不与 patch 来源论文同题、同附件或共享关键结论；
7. 测试记录、闭环摘要和训练日志完整。

`regression_verified` 可以进入对应 Runtime Profile 的正式导出包，但不等于已有比赛证据。

## 4. competition_evidenced

在 `regression_verified` 基础上，还必须具备失败修复重测、正式比赛记录、唯一实验组、证据内容哈希和摘要绑定的人工批准。该状态只表示已有正式比赛证据，不宣称 Patch 普遍正确或稳定获奖。

## 5. plugin 提案

同一类题中至少 3 篇不同论文、不同题号反复出现同一经验，并且该经验通过旧题闭环验证后，可以提出 plugin candidate。

例子：

```text
先建立评价函数，再转化为优化问题
```

这类经验可能适合工程优化 plugin，因为它不依赖单篇论文背景。

不能升级的例子：

```text
同心圆布局降维
```

它只适合特定空间布局背景，应留在 A092/A127 patch 中。

## 6. plugin 正式规则

plugin candidate 满足以下条件后，才允许建议写入正式 plugin：

1. 通过至少 3 道 T3/T4 官方旧题；
2. 覆盖至少 2 个不同背景；
3. 对比旧版本没有引入新的 P3/P4/P5；
4. 有失败复盘证明该规则解决的是稳定问题，而不是单题偶然问题；
5. 人工确认升级范围。

## 7. base 提案

只有跨题型也成立的经验，才允许提出 base candidate。

可考虑进入 base 的经验通常是：

1. 先诊断，后建模；
2. 先写输入、输出、变量、指标和约束；
3. 先解释模型适用性，再使用模型；
4. 没有人工确认，不进入下一阶段。

不得进入 base 的经验通常是：

1. 某个题型特有的模型路线；
2. 某篇论文的具体公式；
3. 某种算法偏好；
4. 某个背景下的参数设置；
5. 只在一个旧题上表现好的表达模板。

## 8. 降级和弃用

出现以下情况时，应降级或回滚：

1. 新 patch 导致不相似旧题乱套；
2. 旧题闭环出现 P3/P4/P5；
3. AI 把不可照搬内容迁移到新题；
4. runtime_pack 变长但诊断质量没有提升；
5. 新版本表现低于旧版本。

降级顺序：

```text
competition_evidenced
-> regression_verified
-> review_ready
-> draft
-> deprecated
```

任何降级都必须写入 `training_log.md` 和对应失败复盘。
