# 2024-C 目标复算错误诊断

日期：2026-07-13

诊断：`2024c_objective_recomputation_diagnosis_v1`

结论：第一轮报告中 R01/R02 的大额目标差异主要来自仓库侧 v1 Validator 的数据预处理错误。修正后，R02 四个目标值全部通过外部复算；R01 仍因把销售上限按作物而不是作物-季次汇总而失败。两个 Run 仍存在轮作约束错误，因此该结论不改变第一轮“正控未通过”的总体判断，但改变了失败归因。

## 1. 诊断方法

沿以下边界逐层比较：

```text
官方附件
-> 数据预处理
-> 决策变量 assignments
-> 候选内部评价器
-> formal_result.json
-> 表格与论文
-> 仓库侧外部 Validator
```

本次使用本机保留的 R01/R02 原始归档，并在诊断 JSON 中绑定正式结果 SHA-256。没有重新求解，也没有修改历史 v1 Validator 或历史实验输出。

## 2. 外部 Validator 的数据错误

### 2.1 合并单元格未向下填充

附件 2 的“2023年的农作物种植情况”工作表把同一地块的多条作物记录合并显示。v1 使用 openpyxl 逐行读取，遇到后续行 `row[0] is None` 时直接跳过，没有继承上一行地块编号。

| 口径 | 2023 种植记录 | 作物-季次销售上限键 |
|---|---:|---:|
| v1，不填充合并地块 | 54 | 31 |
| v2，向下填充 | 87 | 47 |

销售上限由 2023 实际面积乘亩产量得到。漏掉 33 条记录会系统性压低正常销售上限，因而把大量正常产量误判为滞销或超额产量。

### 2.2 智慧大棚第一季参数来源错误

附件 1 说明智慧大棚第一季约为 3–7 月，第二季约为 8 月至次年 2 月；智慧大棚主要在冬季自动调温。附件 2 只显式列出智慧大棚第二季统计参数。v1 把第二季参数复制给第一季，R01/R02 则把普通大棚第一季参数用于智慧大棚第一季。

v2 固定为后者。该口径与生长季说明及 2023 第一季实际种植记录一致，且能与 R02 的四个内部目标逐项对齐。

### 2.3 价格聚合不是根因

官方附件中，同一作物-季次在不同适宜地块类型上的销售价格一致。取首值与取平均值结果相同，因此价格聚合没有造成此次目标差异。v2 仍显式检查唯一性，发现不一致时拒绝复算。

## 3. 目标差异复盘

### R02 Treatment

| 场景 | v1 目标差/元 | v2 目标差/元 | v2 目标结论 |
|---|---:|---:|---|
| q1_waste | 12,526,498.667 | 0 | 通过 |
| q1_discount | 5,290,201.500 | 4.84e-7 | 通过 |
| q2_frozen | 14,383,320.757 | 4.99e-7 | 通过 |
| q3_frozen | 14,383,320.757 | 4.99e-7 | 通过 |

R02 的 `objective_reported`、候选内部 `evaluate_solution()`、正式 JSON 和论文表格之间没有序列化或抄写分叉。原外部复算失败是 v1 数据预处理造成的假阴性。

### R01 Baseline

| 场景 | v1 目标差/元 | v2 目标差/元 | v2 目标结论 |
|---|---:|---:|---|
| q1_waste | 15,694,728.417 | 1,067,568.000 | 失败 |
| q1_discount | 4,956,082.243 | 215,968.574 | 失败 |
| q2_frozen | 15,384,089.404 | 1,267,654.332 | 失败 |
| q3_frozen | 15,364,313.696 | 1,264,491.736 | 失败 |

R01 修正外部数据后仍不一致。其内部代码把 2023 销售量汇总为 `demand[crop]`，并让同一作物的不同季次共享年度销售上限；题面要求比较“每种作物每季的总产量”和相应预期销售量，因此冻结合同使用 `(crop_id, season)`。

R01 的正式 JSON 和论文忠实反映了这个内部评价器，没有额外序列化或论文抄写错误。首次分叉发生在目标函数实现的销售上限键。

## 4. 约束错误

目标通过不等于方案可行。v2 按智慧大棚实际季次顺序检查：

```text
2023 第一季 -> 2023 第二季 -> 2024 第一季 -> 2024 第二季 -> ...
```

R01 四个场景各有 1–2 个违约，主要是遗漏 2023 第二季到 2024 第一季边界。R02 的 `q1_waste` 无硬约束违约，其余场景各有 8–10 个违约；R02 只比较同名季次的相邻年份，既会漏掉同年第一季到第二季，也会漏掉第二季到下一年第一季。

因此：

- R02 的目标值可复算，不代表 R02 方案整体有效；
- R01 的目标与约束均未通过；
- 第一轮正控仍不能作为 A092 晋级证据。

## 5. 分层归因

| 层级 | R01 | R02 |
|---|---|---|
| 模型公式 | 销售上限分组错误 | 冻结目标口径通过 |
| 数据预处理 | 候选正确；v1 外部错误 | 候选正确；v1 外部错误 |
| 决策变量解释 | assignments 字段可被 v2 直接读取 | assignments 字段可被 v2 直接读取 |
| 结果序列化 | 未发现分叉 | 未发现分叉 |
| 目标函数实现 | 按作物聚合，错误 | 按作物-季次聚合，正确 |
| 论文抄写 | 忠实抄写内部错误值 | 忠实抄写内部正确目标值 |
| 约束实现 | 2023 边界遗漏 | 实际季次序列错误 |

## 6. 对 A092 v2 的直接要求

1. 外部 Validator 自身必须有附件预处理回归测试，不能只测试公式函数。
2. 对合并单元格、前向填充、缺失参数补全和聚合键生成审计摘要。
3. 目标函数必须输出销售上限键的定义，并用至少一个跨季次小例子验证。
4. 轮作约束必须显式列出时间槽顺序和 2023 到 2024 的边界测试。
5. 自检和外部复算必须使用不同实现；同一 `evaluate_solution()` 的二次调用不算独立验证。
6. 论文强结论必须同时满足目标复算和硬约束检查，不能因目标通过而忽略可行性失败。
