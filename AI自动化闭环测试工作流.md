# AI 自动化闭环测试工作流

## 1. 文档定位

本文件用于指导 AI 代理自动完成数学建模提示词系统的闭环测试链路。

闭环测试不是让 AI 直接解题、写代码或写论文，而是自动完成以下链路：

```text
读取训练材料
-> 选择旧题与提示词组合
-> 运行总控诊断
-> 按评分表自评和机评
-> 识别失败标签
-> 生成旧题测试记录
-> 必要时生成失败复盘卡
-> 提出提示词局部修复方案
-> 更新训练日志和版本日志
-> 生成下一轮重测任务
-> 达标后标记候选稳定版
```

核心目标是让 AI 能够在最少人工干预下完成一次可追踪、可复盘、可回滚的提示词测试循环。

## 2. 自动化边界

### 2.1 AI 自动执行

AI 可以自动执行：

1. 扫描项目目录和已有训练材料；
2. 读取 base、plugin、patch、学习卡片和知识卡片；
3. 根据旧题材料生成总控诊断；
4. 按 100 分评分表打分并说明扣分原因；
5. 检查 P1-P10 失败标签；
6. 生成旧题测试记录文件；
7. 生成失败复盘卡；
8. 判断应修改 base、plugin 还是 patch；
9. 生成局部提示词补丁建议；
10. 更新 `training_log.md`；
11. 更新 `prompt_versions/prompt_version_log.md`；
12. 生成下一轮重测任务；
13. 汇总当前闭环状态。

### 2.2 必须人工确认

以下节点必须暂停等待人工确认：

1. 是否允许把 AI 生成的提示词补丁写入正式文件；
2. 是否允许将某个版本标记为 `stable`；
3. 是否允许把单篇论文经验提升到 `prompt_base`；
4. 是否进入论文写作、代码实现或正式新题执行阶段；
5. 当旧题材料不完整时，是否允许 AI 使用合理假设补全。

### 2.3 禁止自动执行

闭环测试阶段禁止 AI 自动执行：

1. 不得直接写完整竞赛论文；
2. 不得直接写求解代码；
3. 不得跳过总控诊断进入模型求解；
4. 不得因为一次旧题通过就标记 `stable`；
5. 不得把单篇论文特殊技巧直接写入 `prompt_base`；
6. 不得大幅重写已有提示词，只允许局部修复；
7. 不得删除旧版本、旧测试记录或失败复盘。

## 4. 自动化闭环总流程

完整流程如下：

```text
S0 项目预检
S1 材料装载
S2 旧题任务建立
S3 总控诊断生成
S4 评分与失败标签检测
S5 闸门判定
S6 测试记录生成
S7 失败复盘生成
S8 提示词局部修复建议
S9 日志与版本记录更新
S10 重测任务生成
S11 稳定性候选判断
```

## 5. S0 项目预检

### 5.1 输入

项目根目录。

### 5.2 自动检查项

AI 必须检查以下文件是否存在：

```text
数模AI训练总控工作流v3.md
training_log.md
prompt_base/prompt_base_v1.0.md
prompt_plugins/plugin_optimization_v1.md
tests/old_problems/旧题测试_工程优化_模板.md
reviews/failure_cards/失败复盘_模板.md
prompt_versions/prompt_version_log.md
```

### 5.3 输出

预检摘要：

```text
项目预检结果：
- 必需文件是否齐全：
- 缺失文件：
- 当前可用 base：
- 当前可用 plugin：
- 当前可用 patch：
- 当前已有学习卡片：
- 当前已有知识卡片：
- 当前稳定版本：
- 是否允许进入闭环测试：
```

### 5.4 失败分支

如果缺少关键文件：

1. 缺少模板：AI 先根据现有总控文档生成模板草案；
2. 缺少 base：停止闭环，要求先建立 base；
3. 缺少 plugin：只运行 base 测试，并标记插件缺失风险；
4. 缺少 patch：只运行 base + plugin 测试；
5. 缺少旧题：生成旧题材料需求清单，等待人工提供。

每次闭环测试至少需要：

```text

知识卡片 JSON
旧题原文
旧题附件说明
```

### 6.2 装载顺序

AI 必须按以下顺序读取材料：

1. `数模AI训练总控工作流v3.md`
2. `training_log.md`
3. `prompt_base/prompt_base_v1.0.md`
4. 对应题型的 plugin
5. 对应论文经验 patch
6. 对应论文学习卡片

9. 旧题测试模板
10. 失败复盘模板

### 6.3 材料摘要格式

AI 装载完成后，必须输出：

```text
本轮材料摘要：
- 题型：
- 旧题：
- base：
- plugin：
- patch：
- 学习卡片：
- 知识卡片：
- 旧题附件：
- 本轮测试目标：
- 本轮禁止事项：
```

## 7. S2 旧题任务建立

### 7.1 旧题选择规则

工程优化类旧题优先选择具备以下特征的题：

1. 存在最大化或最小化目标；
2. 存在工程、物理、空间、资源或机制约束；
3. 需要定义目标函数、评价函数或综合指标；
4. 决策变量和约束较多；
5. 需要图表解释结果；
6. 可以检验 AI 是否滥用高级模型。

### 7.2 任务卡格式

AI 必须先生成任务卡：

```text
# 闭环测试任务卡

- 测试日期：
- 旧题年份与题号：
- 题型：
- 使用 base：
- 使用 plugin：
- 使用 patch：
- 参考学习卡片：
- 参考知识卡片：
- 测试模块：总控诊断
- 是否写论文：否
- 是否写代码：否
- 是否直接给最终答案：否
- 本轮重点检查：
  1. 是否先建立评价函数或目标函数；
  2. 是否明确目标、变量、约束、评价指标；
  3. 是否比较简单方法和高级优化算法；
  4. 是否给出数据需求；
  5. 是否给出图表计划；
  6. 是否给出人工确认项。
```

## 8. S3 总控诊断生成

### 8.1 执行原则

AI 只生成总控诊断，不写论文，不写代码，不直接给最终解。

### 8.2 必须输出的结构

总控诊断必须包含：

```text
一、训练材料理解
二、旧题总控诊断
三、优化类专项检查
四、模型候选与模型质疑
五、数据需求与代码准备
六、图表与结果计划
七、人工确认项
八、100 分自评
九、失败标签检查
十、最终判定
```

### 8.3 工程优化硬规则

如果旧题属于工程优化类，必须检查：

1. 是否先建立评价函数或目标函数；
2. 是否先拆工程机制；
3. 是否明确优化对象；
4. 是否明确优化目标；
5. 是否明确决策变量；
6. 是否明确约束条件；
7. 是否明确评价指标；
8. 是否比较简单方法与智能优化算法；
9. 是否考虑降维、分区、规则化或参数化；
10. 是否规划优化前后对比和敏感性分析。

### 8.4 自动化输出位置

总控诊断全文应写入闭环摘要文件：

```text
output/closed_loop/闭环摘要_{题型}_{年份题号}_{日期}.md
```

## 9. S4 评分与失败标签检测

### 9.1 100 分评分表

AI 必须按以下评分表打分：

| 项目 | 分值 | 自动评分依据 |
|---|---:|---|
| 题目理解 | 15 | 是否抓住背景、核心矛盾、已知未知条件 |
| 子问题拆解 | 15 | 是否拆清每一问的表面任务和数学任务 |
| 题型判断 | 10 | 是否正确识别评价、预测、优化、仿真等题型 |
| 模型候选与取舍 | 20 | 是否给出候选模型、适用原因、局限和替代方案 |
| 数据需求 | 10 | 是否列出数据项、来源、用途、缺失处理和风险 |
| 建模路线 | 15 | 是否形成输入、处理、输出链路 |
| 图表与结果计划 | 10 | 图表是否服务结论，是否覆盖对比和敏感性分析 |
| 人工确认项 | 5 | 是否列出进入下一阶段前必须确认的问题 |

### 9.2 关键失败标签

AI 必须检查以下标签：

| 标签 | 含义 | 关键程度 |
|---|---|---|
| P1 | 题目理解错误 | 高 |
| P2 | 子问题拆解错误 | 高 |
| P3 | 模型不匹配 | 最高 |
| P4 | 高级模型滥用 | 最高 |
| P5 | 数据需求不清 | 最高 |
| P6 | 代码不可执行 | 中，本阶段通常不触发 |
| P7 | 结果只复述数字 | 中 |
| P8 | 图表不服务论证 | 中 |
| P9 | 摘要空泛 | 低，本阶段通常不触发 |
| P10 | 审稿没发现关键问题 | 中 |

### 9.3 自动判定规则

```text
总分 >= 80 且无 P3/P4/P5：
  通过总控诊断，可以进入下一阶段候选。

总分 >= 80 但出现 P3/P4/P5：
  不通过，必须失败复盘。

总分 60-79：
  不通过，局部修改提示词后重测。

总分 < 60：
  不通过，重构总控诊断提示词或题型插件。
```

## 10. S5 闸门判定

AI 必须输出闸门判定：

```text
闸门判定：
- 总分：
- 是否 >= 80：
- 是否出现 P3/P4/P5：
- 是否允许进入下一阶段：
- 是否必须生成失败复盘：
- 是否建议修改提示词：
- 建议修改层级：base / plugin / patch / 不修改
- 是否需要人工确认：
```

## 11. S6 旧题测试记录生成

### 11.1 生成规则

每次测试都必须生成一份旧题测试记录。

文件位置：

```text
tests/old_problems/旧题测试_{题型}_{年份题号}_{日期}.md
```

### 11.2 内容结构

测试记录必须包含：

```text
# 旧题测试_{题型}_{年份题号}_{日期}

## 1. 基本信息

## 2. 输入材料

## 3. 总控诊断输出摘要

## 4. 100 分评分表

## 5. 工程优化 MVP 必查项

## 6. 失败标签

## 7. 闸门判定

## 8. 下一步动作
```

### 11.3 摘要要求

测试记录中不必粘贴完整总控诊断，但必须保留：

1. 关键建模路线；
2. 候选模型和取舍；
3. 数据需求摘要；
4. 图表计划摘要；
5. 人工确认项；
6. 扣分原因；
7. 失败标签。

完整诊断放在 `output/closed_loop/`。

## 12. S7 失败复盘生成

### 12.1 触发条件

满足任一条件时，必须生成失败复盘：

1. 总分低于 80；
2. 出现 P3；
3. 出现 P4；
4. 出现 P5；
5. AI 直接写论文、写代码或给最终答案；
6. 总控诊断缺少人工确认项；
7. 模型选择明显无法服务题目；
8. 数据需求无法支撑模型。

### 12.2 文件位置

```text
reviews/failure_cards/失败复盘_{题型}_{年份题号}_{日期}.md
```

### 12.3 复盘内容

失败复盘必须回答：

```text
1. 失败现象是什么；
2. 失败标签是什么；
3. 问题来自 base、plugin、patch、AI 执行还是旧题材料；
4. 哪一句提示词约束不够；
5. 应该新增什么硬规则；
6. 应该修改哪个文件；
7. 修改后如何重测；
8. 是否允许进入 stable。
```

## 13. S8 提示词局部修复建议

### 13.1 修复层级判断

AI 必须按以下规则判断修改层级：

| 问题类型 | 修改层级 |
|---|---|
| 所有题型都应遵守的基本闸门缺失 | base |
| 某一题型通用规则缺失 | plugin |
| 单篇论文经验迁移不到位 | patch |
| 只是本次输出执行不到位 | 不改提示词，重测 |
| 旧题材料不完整 | 不改提示词，补材料 |

### 13.2 base 修改限制

只有满足以下条件之一，才允许建议修改 base：

1. 同一问题在 3 个以上不同题型测试中反复出现；
2. 缺失的是通用工作流闸门；
3. 缺失的是通用人工确认项；
4. 缺失的是通用数据降级规则；
5. 人工明确要求提升到 base。

### 13.3 plugin 修改条件

如果问题只属于某一题型，例如优化类，则优先修改 plugin：

1. 优化目标不清；
2. 决策变量不清；
3. 约束条件不清；
4. 高级优化算法滥用；
5. 缺少简单方法优先检查；
6. 缺少敏感性分析。

### 13.4 patch 修改条件

如果问题只与某篇论文经验有关，则修改 patch：

1. A092 的评价函数先行经验没有被触发；
2. A127 的分区、分层、规则化经验没有被触发；
3. AI 照搬了论文中的特殊公式、布局或数值；
4. AI 没有说明哪些经验不可迁移。

### 13.5 修复草案格式

AI 生成修复草案时必须使用以下格式：

```text
# 提示词修复草案_{题型}_{年份题号}_{日期}

## 1. 触发原因

## 2. 对应失败标签

## 3. 建议修改层级

## 4. 建议修改文件

## 5. 原提示词漏洞

## 6. 建议新增片段

待插入的提示词片段：

    在这里写入建议新增或替换的提示词片段。

## 7. 不建议修改的内容

## 8. 重测标准
```

注意：修复草案默认写入 `output/closed_loop/`，不得直接覆盖正式提示词文件。只有人工确认后，才允许写入 `prompt_base/`、`prompt_plugins/` 或 `prompt_patches/`。

## 14. S9 日志与版本记录更新

### 14.1 更新 training_log.md

每次闭环测试必须在 `training_log.md` 增加一行：

```text
| 日期 | 论文/旧题 | 使用版本 | 测试模块 | 得分 | 失败标签 | 修改内容 | 是否稳定 |
```

示例：

```text
| 2026-07-02 | 2021-B 工程优化旧题 | prompt_base_v1.0 + plugin_optimization_v1 + patch_A092_engineering_optimization | 总控诊断 | 76 | P5 | 生成失败复盘，建议补强数据需求表 | 否 |
```

### 14.2 更新失败标签统计

如果出现失败标签，AI 必须同步更新标签统计：

```text
| 标签 | 出现次数 | 最近一次出现 | 主要原因 | 下一步动作 |
```

### 14.3 更新稳定版本记录

只有当同类旧题至少通过 2 次，且没有 P3/P4/P5，才允许建议更新稳定版本记录。

### 14.4 更新 prompt_version_log.md

如果人工确认了正式提示词修改，AI 必须新增版本记录：

```text
| 日期 | 版本 | 层级 | 状态 | 来源论文/旧题 | 引入经验 | 修改模块 | 测试题 | 得分 | 失败标签 | 是否稳定 | 备注 |
```

状态规则：

```text
draft：刚生成，未测试
testing：已进入旧题测试
stable：至少通过 2 道同类旧题
rollback：已回滚
frozen：赛前冻结
```

## 15. S10 重测任务生成

### 15.1 触发条件

以下情况必须生成重测任务：

1. 修改了任何提示词；
2. 总分低于 80；
3. 出现 P3/P4/P5；
4. 旧题材料补充后需要重跑；
5. AI 输出执行不到位但提示词不需要修改。

### 15.2 重测任务格式

```text
# 重测任务_{题型}_{年份题号}_{日期}

## 1. 重测原因

## 2. 上轮失败标签

## 3. 本轮使用版本

## 4. 本轮重点检查

## 5. 必须改善项

## 6. 通过标准

## 7. 如果仍失败
```

### 15.3 重测通过标准

工程优化类重测必须满足：

1. 总控诊断得分 >= 80；
2. 不出现 P3/P4/P5；
3. 明确评价函数或目标函数；
4. 明确优化目标、决策变量、约束条件和评价指标；
5. 比较简单方法和智能优化算法；
6. 有数据需求表；
7. 有图表计划；
8. 有敏感性分析计划；
9. 有人工确认项。

## 16. S11 稳定性候选判断

### 16.1 候选稳定版条件

同一题型满足以下条件时，AI 可以建议标记为候选稳定版：

1. 至少通过 2 道同类旧题；
2. 两次总控诊断均 >= 80；
3. 两次均未出现 P3/P4/P5；
4. 输出结构稳定；
5. 人工确认项完整；
6. 数据需求清楚；
7. 模型选择没有滥用高级算法；
8. 图表计划能支撑结论。

### 16.2 不允许稳定的情况

出现以下任一情况，不得标记 `stable`：

1. 只通过 1 道旧题；
2. 通过但出现 P3/P4/P5；
3. 通过依赖大量人工补救；
4. 测试题型与目标题型不一致；
5. 只是 patch 有效，base 或 plugin 仍不稳定；
6. 新版低于旧版；
7. 缺少测试记录或失败复盘。

## 18. 自动化执行状态机

```text
INIT
  -> PREFLIGHT
  -> LOAD_MATERIALS
  -> BUILD_TASK
  -> RUN_DIAGNOSIS
  -> SCORE
  -> GATE

GATE:
  if score >= 80 and no P3/P4/P5:
      -> WRITE_TEST_RECORD
      -> UPDATE_LOG
      -> CHECK_STABLE_CANDIDATE
      -> DONE

  if score < 80 or has P3/P4/P5:
      -> WRITE_TEST_RECORD
      -> WRITE_FAILURE_CARD
      -> WRITE_FIX_DRAFT
      -> UPDATE_LOG
      -> WRITE_RETEST_TASK
      -> WAIT_HUMAN_CONFIRMATION

WAIT_HUMAN_CONFIRMATION:
  if fix approved:
      -> APPLY_FIX
      -> UPDATE_VERSION_LOG
      -> RETEST

  if fix rejected:
      -> WRITE_RETEST_TASK_WITH_NO_PROMPT_CHANGE
      -> RETEST

RETEST:
  -> LOAD_MATERIALS
  -> RUN_DIAGNOSIS
```

## 21. 本工作流的完成标准

一次自动化闭环完成，必须产出：

1. 旧题测试记录；
2. 闭环摘要；
3. 训练日志更新；
4. 闸门判定；
5. 如失败，必须有失败复盘；
6. 如需修改，必须有修复草案；
7. 如需重测，必须有重测任务；
8. 最终说明是否可以进入下一阶段。

当同一题型至少完成 2 次通过闭环后，才允许进入稳定版评估。
